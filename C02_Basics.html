2. 基础

本章旨在介绍 Lean 中数学推理的基本环节：计算、应用引理和定理，以及对一般结构进行推理。

2.1. 计算

我们通常学会进行数学计算时，并不会把这些计算看作证明。但是，当我们像 Lean 要求的那样，为计算中的每一步给出理由时，最终得到的正是一个证明：它证明该计算左端等于右端。

在 Lean 中，陈述一个定理等同于陈述一个目标，也就是证明该定理这一目标。Lean 提供重写策略 rw，用于在目标中把某个恒等式的左端替换为右端。若 a、b 和 c 是实数，则 mul_assoc a b c 是恒等式 a * b * c = a * (b * c)，而 mul_comm a b 是恒等式 a * b = b * a。Lean 提供的自动化通常使我们不必显式引用这类事实，但它们作为说明材料很有用。在 Lean 中，乘法向左结合，因此 mul_assoc 的左端也可以写作 (a * b) * c。不过，通常更好的风格是注意 Lean 的记号约定：Lean 省略括号时，我们也相应省略。

让我们试用一下 rw。

example (a b c : ℝ) : a * b * c = b * (a * c) := by
  rw [mul_comm a b]
  rw [mul_assoc b a c]

关联例子文件开头的 import 行，从 Mathlib 导入了实数理论以及有用的自动化工具。为了简洁，本教材通常省略这类信息。

欢迎你修改代码，观察会发生什么。在 VS Code 中，可以用 \R 或 \real 输入字符 ℝ。只有在按下空格键或 Tab 键之后，该符号才会出现。阅读 Lean 文件时，如果把鼠标悬停在某个符号上，VS Code 会显示可用于输入该符号的语法。如果想查看所有可用缩写，可以按 Ctrl-Shift-P，然后输入 abbreviations，以打开 Lean 4: Show Unicode Input Abbreviations 命令。如果键盘上不便输入反斜杠，也可以修改 lean4.input.leader 设置来改变前导字符。

当光标位于一个策略证明中间时，Lean 会在 Lean Infoview 窗口中报告当前的证明状态。随着你把光标逐步移过证明中的每一步，可以看到状态发生变化。Lean 中一个典型的证明状态可能如下所示：

1 goal
x y : ℕ,
h₁ : Prime x,
h₂ : ¬Even x,
h₃ : y > x
⊢ y ≥ 4

以 ⊢ 开头的那一行之前的各行表示语境：它们是当前正在讨论的对象和假设。在这个例子中，语境包括两个对象 x 和 y，它们都是自然数；还包括三个假设，分别标记为 h₁、h₂ 和 h₃。在 Lean 中，语境中的一切都有一个标识符。你可以将这些带下标的标号输入为 h\1、h\2 和 h\3，但任何合法标识符都可以；例如也可以使用 h1、h2、h3，或者 foo、bar 和 baz。最后一行表示目标，也就是待证明的事实。有时人们用靶标指待证明的事实，而用目标指语境与靶标的组合。在实践中，具体含义通常由上下文决定。

请尝试证明这些恒等式；每一次都把 sorry 替换为一个策略证明。使用 rw 策略时，可以用左箭头（\l）反向使用一个恒等式。例如，rw [← mul_assoc a b c] 会在当前目标中把 a * (b * c) 替换为 a * b * c。注意，左箭头表示按 mul_assoc 给出的恒等式从右向左使用；它与目标本身的左端或右端无关。

example (a b c : ℝ) : c * b * a = b * (a * c) := by
  sorry

example (a b c : ℝ) : a * (b * c) = b * (a * c) := by
  sorry

你也可以不带参数地使用 mul_assoc、mul_comm 这样的恒等式。在这种情况下，重写策略会试图用恒等式的左端匹配目标中的某个表达式，并采用它找到的第一个匹配模式。

example (a b c : ℝ) : a * b * c = b * c * a := by
  rw [mul_assoc]
  rw [mul_comm]

你也可以只提供部分信息。例如，mul_comm a 会匹配任何形如 a * ? 的模式，并将其重写为 ? * a。请尝试在第一个例子中完全不提供参数，在第二个例子中只提供一个参数。

example (a b c : ℝ) : a * (b * c) = b * (c * a) := by
  sorry

example (a b c : ℝ) : a * (b * c) = b * (a * c) := by
  sorry

也可以把 rw 用于局部语境中的事实。

example (a b c d e f : ℝ) (h : a * b = c * d) (h' : e = f) : a * (b * e) = c * (d * f) := by
  rw [h']
  rw [← mul_assoc]
  rw [h]
  rw [mul_assoc]

请尝试下面的例子；第二个例子可使用定理 sub_self：

example (a b c d e f : ℝ) (h : b * c = e * f) : a * b * c * d = a * e * f * d := by
  sorry

example (a b c d : ℝ) (hyp : c = b * a - d) (hyp' : d = a * b) : c = 0 := by
  sorry

多个重写命令可以合并为一个命令：只需在方括号中用逗号列出相关恒等式即可。

example (a b c d e f : ℝ) (h : a * b = c * d) (h' : e = f) : a * (b * e) = c * (d * f) := by
  rw [h', ← mul_assoc, h, mul_assoc]

把光标放在任意重写列表中的逗号之后，仍然可以看到逐步推进的中间状态。

另一个技巧是，可以在例子或定理之外一次性声明变量。Lean 随后会自动把这些变量纳入相关语境。

variable (a b c d e f : ℝ)

example (h : a * b = c * d) (h' : e = f) : a * (b * e) = c * (d * f) := by
  rw [h', ← mul_assoc, h, mul_assoc]

检查上述证明开头的策略状态，可以看到 Lean 的确纳入了所有变量。我们可以把声明放进 section ... end 块中，以限定其作用域。最后，回忆引言中提到的内容：Lean 为我们提供了一个命令，用于确定表达式的类型。

section
variable (a b c : ℝ)

#check a
#check a + b
#check (a : ℝ)
#check mul_comm a b
#check (mul_comm a b : a * b = b * a)
#check mul_assoc c a b
#check mul_comm a
#check mul_comm

end

#check 命令既适用于对象，也适用于事实。对于命令 #check a，Lean 会报告 a 的类型是 ℝ。对于命令 #check mul_comm a b，Lean 会报告 mul_comm a b 是事实 a * b = b * a 的一个证明。命令 #check (a : ℝ) 表明我们预期 a 的类型是 ℝ；若事实并非如此，Lean 会报错。最后三个 #check 命令的输出我们稍后会解释；在此之前，你可以先查看它们，并自行尝试一些 #check 命令。

再看几个例子。定理 two_mul a 表示 2 * a = a + a。定理 add_mul 和 mul_add 表达乘法对加法的分配律，定理 add_assoc 表达加法的结合律。请使用 #check 命令查看它们的精确陈述。

example : (a + b) * (a + b) = a * a + 2 * (a * b) + b * b := by
  rw [mul_add, add_mul, add_mul]
  rw [← add_assoc, add_assoc (a * a)]
  rw [mul_comm b a, ← two_mul]

虽然可以在编辑器中逐步查看这个证明，从而弄清它究竟在做什么，但单独阅读它并不容易。Lean 提供了一种更结构化的方式，用关键字 calc 来书写这类证明。

example : (a + b) * (a + b) = a * a + 2 * (a * b) + b * b :=
  calc
    (a + b) * (a + b) = a * a + b * a + (a * b + b * b) := by
      rw [mul_add, add_mul, add_mul]
    _ = a * a + (b * a + a * b) + b * b := by
      rw [← add_assoc, add_assoc (a * a)]
    _ = a * a + 2 * (a * b) + b * b := by
      rw [mul_comm b a, ← two_mul]

注意，这个证明并不以 by 开头：以 calc 开头的表达式是一个证明项。calc 表达式也可以用于策略证明内部，但 Lean 会把它解释为一条指令：使用由此得到的证明项来解决当前目标。calc 语法比较严格：下划线和论证必须采用上面展示的格式。Lean 还会使用缩进来判断策略块或 calc 块从何处开始、到何处结束；可以尝试改变上面证明的缩进，看看会发生什么。

书写 calc 证明的一种方法，是先用 sorry 策略作为论证占位，勾勒出证明框架，确认 Lean 在这些占位下接受该表达式，然后再用策略逐步证明各个步骤。

example : (a + b) * (a + b) = a * a + 2 * (a * b) + b * b :=
  calc
    (a + b) * (a + b) = a * a + b * a + (a * b + b * b) := by
      sorry
    _ = a * a + (b * a + a * b) + b * b := by
      sorry
    _ = a * a + 2 * (a * b) + b * b := by
      sorry

请尝试用一个纯粹的 rw 证明，以及一个更结构化的 calc 证明，来证明下面的恒等式：

example : (a + b) * (c + d) = a * c + a * d + b * c + b * d := by
  sorry

下面的练习稍有挑战。你可以使用列在其后的定理。

example (a b : ℝ) : (a + b) * (a - b) = a ^ 2 - b ^ 2 := by
  sorry

#check pow_two a
#check mul_sub a b c
#check add_mul a b c
#check add_sub a b c
#check sub_sub a b c
#check add_zero a

我们也可以在语境中的某个假设里进行重写。例如，rw [mul_comm a b] at hyp 会在假设 hyp 中把 a * b 替换为 b * a。

example (a b c d : ℝ) (hyp : c = d * a + b) (hyp' : b = a * d) : c = 2 * a * d := by
  rw [hyp'] at hyp
  rw [mul_comm d a] at hyp
  rw [← two_mul (a * d)] at hyp
  rw [← mul_assoc 2 a d] at hyp
  exact hyp

在最后一步，exact 策略可以使用 hyp 解决目标，因为此时 hyp 与目标完全匹配。

本节最后指出，Mathlib 通过 ring 策略提供了一项有用的自动化。只要某个恒等式纯粹由环公理推出，而不使用任何局部假设，该策略就被设计用来在任意交换环中证明它。

example : c * b * a = b * (a * c) := by
  ring

example : (a + b) * (a + b) = a * a + 2 * (a * b) + b * b := by
  ring

example : (a + b) * (a - b) = a ^ 2 - b ^ 2 := by
  ring

example (hyp : c = d * a + b) (hyp' : b = a * d) : c = 2 * a * d := by
  rw [hyp, hyp']
  ring

当我们导入 Mathlib.Data.Real.Basic 时，ring 策略会被间接导入；不过下一节会看到，它也可用于实数以外的结构上的计算。它也可以通过命令 import Mathlib.Tactic 显式导入。我们还会看到，对于其他常见的代数结构，也存在类似的策略。

rw 有一个变体叫 nth_rw，允许你只替换目标中某个表达式的特定出现。可能的匹配从 1 开始编号；因此在下面的例子中，nth_rw 2 [h] 会把第二次出现的 a + b 替换为 c。

example (a b c : ℕ) (h : a + b = c) : (a + b) * (a + b) = a * c + b * c := by
  nth_rw 2 [h]
  rw [add_mul]

2.2. 证明代数结构中的恒等式

从数学上说，一个环由一族对象 \(R\)、运算 \(+\) 与 \(\times\)、常元 \(0\) 和 \(1\)，以及运算 \(x \mapsto -x\) 组成，并满足如下条件：

  • \(R\) 连同 \(+\) 构成一个阿贝尔群，其中 \(0\) 是加法单位元，取负给出逆元。
  • 乘法满足结合律，以 \(1\) 为单位元，并且乘法对加法满足分配律。

在 Lean 中，这族对象表示为一个类型 R。环公理如下：

variable (R : Type*) [Ring R]

#check (add_assoc : ∀ a b c : R, a + b + c = a + (b + c))
#check (add_comm : ∀ a b : R, a + b = b + a)
#check (zero_add : ∀ a : R, 0 + a = a)
#check (neg_add_cancel : ∀ a : R, -a + a = 0)
#check (mul_assoc : ∀ a b c : R, a * b * c = a * (b * c))
#check (mul_one : ∀ a : R, a * 1 = a)
#check (one_mul : ∀ a : R, 1 * a = a)
#check (mul_add : ∀ a b c : R, a * (b + c) = a * b + a * c)
#check (add_mul : ∀ a b c : R, (a + b) * c = a * c + b * c)

第一行中的方括号稍后还会详细解释；目前只需知道，该声明给出了一个类型 R，以及 R 上的一个环结构。随后，Lean 允许我们对 R 的元素使用一般的环记号，并使用关于环的定理库。

有些定理的名称应该看起来很熟悉：它们正是上一节中我们用来对实数进行计算的定理。Lean 不仅适合证明关于自然数、整数等具体数学结构的事实，也适合证明关于由公理刻画的抽象结构（例如环）的事实。此外，Lean 支持对抽象结构和具体结构进行一般性推理，并且可以被训练为识别适当的实例。因此，任何关于环的定理都可以应用于整数 ℤ、有理数 ℚ、复数 ℂ 等具体环。它也可以应用于任何扩展环的抽象结构实例，例如任意有序环或任意域。

不过，并非实数的重要性质都在任意环中成立。例如，实数乘法是交换的，但一般并非如此。如果你学过线性代数，就会知道，对每个 \(n\)，实数上的 \(n\) 阶方阵构成一个环，而其中乘法通常不交换。若我们声明 R 是一个交换环，那么上一节中的所有定理，在把 ℝ 替换为 R 后事实上仍然成立。

variable (R : Type*) [CommRing R]
variable (a b c d : R)

example : c * b * a = b * (a * c) := by ring

example : (a + b) * (a + b) = a * a + 2 * (a * b) + b * b := by ring

example : (a + b) * (a - b) = a ^ 2 - b ^ 2 := by ring

example (hyp : c = d * a + b) (hyp' : b = a * d) : c = 2 * a * d := by
  rw [hyp, hyp']
  ring

我们把检查其他所有证明是否原样通过留给读者。注意，当证明很短，例如 by ring、by linarith 或 by sorry 时，通常也允许把它和 by 写在同一行。良好的证明书写风格应当在简洁性与可读性之间取得平衡。

本节的目标是强化你在上一节形成的技能，并把这些技能应用于关于环的公理化推理。我们将从上面列出的公理出发，用它们推出其他事实。我们证明的大多数事实已经存在于 Mathlib 中。为了帮助你熟悉库的内容以及命名约定，我们会给自己证明的版本使用相同名称。

Lean 提供了一种类似程序语言中的组织机制：当一个定义或定理 foo 在命名空间 bar 中引入时，它的全名是 bar.foo。之后命令 open bar 会打开该命名空间，使我们可以使用较短的名称 foo。为了避免名称冲突导致错误，在下一个例子中，我们把自己版本的库定理放进一个名为 MyRing. 的新命名空间。

下一个例子表明，我们不必把 add_zero 或 add_neg_cancel 作为环公理，因为它们可以由其他公理推出。

namespace MyRing
variable {R : Type*} [Ring R]

theorem add_zero (a : R) : a + 0 = a := by rw [add_comm, zero_add]

theorem add_neg_cancel (a : R) : a + -a = 0 := by rw [add_comm, neg_add_cancel]

#check MyRing.add_zero
#check add_zero

end MyRing

其净效果是：我们可以暂时重新证明库中的一个定理，随后继续使用库中的版本。不过不要作弊！在接下来的练习中，请注意只使用本节前面已经证明的一般环论事实。

（如果你非常仔细，可能已经注意到我们把 (R : Type*) 中的圆括号改成了 {R : Type*} 中的花括号。这把 R 声明为一个隐式参数。我们马上会解释这意味着什么；暂时不必担心。）

下面是一个有用的定理：

theorem neg_add_cancel_left (a b : R) : -a + (a + b) = b := by
  rw [← add_assoc, neg_add_cancel, zero_add]

请证明对应的伴随版本：

theorem add_neg_cancel_right (a b : R) : a + b + -b = a := by
  sorry

用这些定理证明如下命题：

theorem add_left_cancel {a b c : R} (h : a + b = a + c) : b = c := by
  sorry

theorem add_right_cancel {a b c : R} (h : a + b = c + b) : a = c := by
  sorry

只要事先规划得当，每一个命题都可以用三次重写完成。

现在我们解释花括号的用法。设想你处在如下情形：语境中有 a、b 和 c，还有假设 h : a + b = a + c，而你想推出结论 b = c。在 Lean 中，定理可以像应用到对象那样应用到假设和事实上，所以你可能会以为 add_left_cancel a b c h 是事实 b = c 的一个证明。但请注意，显式写出 a、b 和 c 是冗余的，因为假设 h 已经清楚表明了我们所考虑的对象。在这个例子中，多敲几个字符并不麻烦；但如果要把 add_left_cancel 应用于更复杂的表达式，写出这些参数就会很繁琐。在这类情况下，Lean 允许我们把参数标记为隐式，意思是这些参数应当省略，并由其他方式推断出来，例如由后续参数和假设推断。{a b c : R} 中的花括号正是这个作用。因此，在给定上述定理陈述后，正确的表达式只是 add_left_cancel h。

作为说明，我们来证明 a * 0 = 0 可由环公理推出。

theorem mul_zero (a : R) : a * 0 = 0 := by
  have h : a * 0 + a * 0 = a * 0 + 0 := by
    rw [← mul_add, add_zero, add_zero]
  rw [add_left_cancel h]

我们使用了一个新技巧！如果逐步查看这个证明，就能看出发生了什么。have 策略引入一个新目标 a * 0 + a * 0 = a * 0 + 0，其语境与原目标相同。下一行缩进这一事实表明，Lean 正期待一个用于证明该新目标的策略块。因此，缩进促成了一种模块化的证明风格：被缩进的子证明建立由 have 引入的目标。之后，我们回到原目标，只是语境中多了一个新假设 h：既然已经证明了它，我们便可以使用它。此时，目标恰好就是 add_left_cancel h 的结果。

我们同样可以用 apply add_left_cancel h 或 exact add_left_cancel h 结束证明。exact 策略的参数是一个能完全证明当前目标的证明项，并且不会产生任何新目标。apply 策略则是一个变体，它的参数不一定是完整证明；缺失的部分要么由 Lean 自动推断，要么成为新的待证明目标。虽然从技术上说 exact 是冗余的，因为它严格弱于 apply，但它能使证明脚本对人类读者稍微更清晰，也更便于在库演化时维护。

请记住，这里并未假定乘法交换，因此下面的定理同样需要一些工作。

theorem zero_mul (a : R) : 0 * a = 0 := by
  sorry

到现在为止，你也应当能够在下一个练习中把每个 sorry 替换为一个证明；仍然只使用本节已经建立的关于环的事实，以及公理 eq_symm。

theorem neg_eq_of_add_eq_zero {a b : R} (h : a + b = 0) : -a = b := by
  sorry

theorem eq_neg_of_add_eq_zero {a b : R} (h : a + b = 0) : a = -b := by
  sorry

theorem neg_zero : (-0 : R) = 0 := by
  apply neg_eq_of_add_eq_zero
  rw [add_zero]

theorem neg_neg (a : R) : - -a = a := by
  sorry

在第三个定理中，我们不得不使用注解 (-0 : R) 而不是 0，因为如果不指定 R，Lean 就无法推断我们指的是哪一个 0，并且默认会把它解释为自然数。

在 Lean 中，环中的减法可证明地等于加上加法逆元。

example (a b : R) : a - b = a + -b :=
  sub_eq_add_neg a b

在实数上，减法正是以这种方式定义的：

example (a b : ℝ) : a - b = a + -b :=
  rfl

example (a b : ℝ) : a - b = a + -b := by
  rfl

证明项 rfl 是 “reflexivity”（自反性）的缩写。把它作为 a - b = a + -b 的证明，会迫使 Lean 展开定义，并识别等式两端其实相同。rfl 策略做的是同一件事。这是 Lean 底层逻辑中所谓定义相等的一个例子。这意味着，不仅可以用 sub_eq_add_neg 进行重写，把 a - b 替换为 a + -b；在某些处理实数的语境中，还可以互换使用等式两端。例如，你现在已经有足够信息证明上一节中的定理 self_sub：

theorem self_sub (a : R) : a - a = 0 := by
  sorry

请证明你可以使用 rw 完成这一点；但如果把任意环 R 替换为实数，也可以使用 apply 或 exact 来证明。

Lean 知道 1 + 1 = 2 在任意环中都成立。稍加努力，你可以用它证明上一节中的定理 two_mul：

theorem one_add_one_eq_two : 1 + 1 = (2 : R) := by
  norm_num

theorem two_mul (a : R) : 2 * a = a + a := by
  sorry

本节最后指出，我们上面建立的某些关于加法和负元的事实，并不需要完整的环公理，甚至不需要加法交换。较弱的概念群可以公理化如下：

variable (A : Type*) [AddGroup A]

#check (add_assoc : ∀ a b c : A, a + b + c = a + (b + c))
#check (zero_add : ∀ a : A, 0 + a = a)
#check (neg_add_cancel : ∀ a : A, -a + a = 0)

约定上，当群运算交换时使用加法记号，否则使用乘法记号。因此，Lean 在加法版本之外也定义了乘法版本（并且还定义了它们的阿贝尔变体 AddCommGroup 和 CommGroup）。

variable {G : Type*} [Group G]

#check (mul_assoc : ∀ a b c : G, a * b * c = a * (b * c))
#check (one_mul : ∀ a : G, 1 * a = a)
#check (inv_mul_cancel : ∀ a : G, a⁻¹ * a = 1)

如果你信心十足，可以尝试只使用这些公理证明下面关于群的事实。在此过程中，你需要证明若干辅助引理。本节已经完成的证明可以提供一些提示。

theorem mul_inv_cancel (a : G) : a * a⁻¹ = 1 := by
  sorry

theorem mul_one (a : G) : a * 1 = a := by
  sorry

theorem mul_inv_rev (a b : G) : (a * b)⁻¹ = b⁻¹ * a⁻¹ := by
  sorry

显式调用这些引理很繁琐，因此 Mathlib 提供了若干类似 ring 的策略来覆盖大多数用法：group 用于非交换乘法群，abel 用于阿贝尔加法群，noncomm_ring 用于非交换环。代数结构称为 Ring 和 CommRing，而策略却命名为 noncomm_ring 和 ring，这看起来或许有些奇怪。这部分出于历史原因，也部分是为了便利：处理交换环的策略使用更频繁，因此给它一个更短的名称。

2.3. 使用定理和引理

重写非常适合用来证明等式；但其他类型的定理又该如何处理？例如，若 \(b \le c\)，如何证明诸如 \(a + e^b \le a + e^c\) 这样的不等式？我们已经看到，定理可以应用于参数和假设，而 apply 与 exact 策略可以用来解决目标。本节将充分利用这些工具。

考虑库中的定理 le_refl 和 le_trans：

#check (le_refl : ∀ a : ℝ, a ≤ a)
#check (le_trans : a ≤ b → b ≤ c → a ≤ c)

正如我们将在 第 3.1 节 中更详细说明的，le_trans 陈述中的隐含括号向右结合，因此它应解释为 a ≤ b → (b ≤ c → a ≤ c)。库的设计者把 le_trans 的参数 a、b 和 c 设为隐式参数，所以 Lean 不会允许你显式提供它们（除非你坚持如此；我们稍后会讨论这一点）。相反，Lean 期望从使用它们的语境中推断这些参数。例如，当语境中有假设 h : a ≤ b 和 h' : b ≤ c 时，下列写法都可行：

variable (h : a ≤ b) (h' : b ≤ c)

#check (le_refl : ∀ a : Real, a ≤ a)
#check (le_refl a : a ≤ a)
#check (le_trans : a ≤ b → b ≤ c → a ≤ c)
#check (le_trans h : b ≤ c → a ≤ c)
#check (le_trans h h' : a ≤ c)

apply 策略接受一个一般命题或蕴含命题的证明，尝试把它的结论与当前目标匹配，并把其中的假设（若有）留下来作为新的目标。如果给出的证明与目标完全匹配（在定义相等意义下），则可以用 exact 策略代替 apply。因此，下面这些写法都成立：

example (x y z : ℝ) (h₀ : x ≤ y) (h₁ : y ≤ z) : x ≤ z := by
  apply le_trans
  · apply h₀
  · apply h₁

example (x y z : ℝ) (h₀ : x ≤ y) (h₁ : y ≤ z) : x ≤ z := by
  apply le_trans h₀
  apply h₁

example (x y z : ℝ) (h₀ : x ≤ y) (h₁ : y ≤ z) : x ≤ z :=
  le_trans h₀ h₁

example (x : ℝ) : x ≤ x := by
  apply le_refl

example (x : ℝ) : x ≤ x :=
  le_refl x

在第一个例子中，应用 le_trans 会产生两个目标，我们用圆点标明每个目标的证明从何处开始。圆点不是必需的，但它们可以聚焦目标：在由圆点引入的代码块中，只显示一个目标，并且必须在该代码块结束前完成它。这里我们通过另起一个圆点块来结束第一个代码块；同样也可以通过减少缩进来达到这一效果。在第三个例子和最后一个例子中，我们完全避免进入策略模式：le_trans h₀ h₁ 与 le_refl x 正是所需的证明项。

下面还有几个库定理：

#check (le_refl : ∀ a, a ≤ a)
#check (le_trans : a ≤ b → b ≤ c → a ≤ c)
#check (lt_of_le_of_lt : a ≤ b → b < c → a < c)
#check (lt_of_lt_of_le : a < b → b ≤ c → a < c)
#check (lt_trans : a < b → b < c → a < c)

请把它们与 apply 和 exact 结合起来，证明如下命题：

example (h₀ : a ≤ b) (h₁ : b < c) (h₂ : c ≤ d) (h₃ : d < e) : a < e := by
  sorry

事实上，Lean 有一个策略可以自动处理这类事情：

example (h₀ : a ≤ b) (h₁ : b < c) (h₂ : c ≤ d) (h₃ : d < e) : a < e := by
  linarith

linarith 策略专门用于处理线性算术。

example (h : 2 * a ≤ 3 * b) (h' : 1 ≤ a) (h'' : d = 2) : d + a ≤ 5 * b := by
  linarith

除了语境中的等式和不等式之外，linarith 还会使用你作为参数传入的额外不等式。在下一个例子中，exp_le_exp.mpr h' 是 exp b ≤ exp c 的一个证明，我们稍后会解释这一点。注意，在 Lean 中，我们写 f x 来表示函数 f 作用于参数 x，这与写 h x 表示事实或定理 h 作用于参数 x 所得结果完全相同。只有复合参数才需要括号，例如 f (x + y)。若没有括号，f x + y 会被解析为 (f x) + y。

example (h : 1 ≤ a) (h' : b ≤ c) : 2 + a + exp b ≤ 3 * a + exp c := by
  linarith [exp_le_exp.mpr h']

下面是库中另外一些可用于建立实数不等式的定理。

#check (exp_le_exp : exp a ≤ exp b ↔ a ≤ b)
#check (exp_lt_exp : exp a < exp b ↔ a < b)
#check (log_le_log : 0 < a → a ≤ b → log a ≤ log b)
#check (log_lt_log : 0 < a → a < b → log a < log b)
#check (add_le_add : a ≤ b → c ≤ d → a + c ≤ b + d)
#check (add_le_add_right : a ≤ b → ∀ c, c + a ≤ c + b)
#check (add_le_add_left : a ≤ b → ∀ c, a + c ≤ b + c)
#check (add_lt_add_of_le_of_lt : a ≤ b → c < d → a + c < b + d)
#check (add_lt_add_of_lt_of_le : a < b → c ≤ d → a + c < b + d)
#check (add_lt_add_right : a < b → ∀ c, c + a < c + b)
#check (add_lt_add_left : a < b → ∀ c, a + c < b + c)
#check (add_nonneg : 0 ≤ a → 0 ≤ b → 0 ≤ a + b)
#check (add_pos : 0 < a → 0 < b → 0 < a + b)
#check (add_pos_of_pos_of_nonneg : 0 < a → 0 ≤ b → 0 < a + b)
#check (exp_pos : ∀ a, 0 < exp a)
#check add_le_add_right

其中一些定理，如 exp_le_exp 和 exp_lt_exp，使用双向蕴含，它表示“当且仅当”。（在 VS Code 中可以用 \lr 或 \iff 输入。）我们将在下一章更详细讨论这个连接词。这类定理可以配合 rw 使用，把目标重写成一个等价目标：

example (h : a ≤ b) : exp a ≤ exp b := by
  rw [exp_le_exp]
  exact h

不过在本节中，我们将使用如下事实：若 h : A ↔ B 是这样的等价命题，则 h.mp 给出正向方向 A → B，而 h.mpr 给出反向方向 B → A。这里 mp 表示 “modus ponens”，mpr 表示 “modus ponens reverse”。如果愿意，也可以分别用 h.1 和 h.2 代替 h.mp 与 h.mpr。因此，下面的证明成立：

example (h₀ : a ≤ b) (h₁ : c < d) : a + exp c + e < b + exp d + e := by
  apply add_lt_add_of_lt_of_le
  · apply add_lt_add_of_le_of_lt h₀
    apply exp_lt_exp.mpr h₁
  apply le_refl

第一行 apply add_lt_add_of_lt_of_le 产生两个目标，我们再次用圆点把第一个目标的证明与第二个目标的证明分开。

请自行尝试下面的例子。中间的例子表明，norm_num 策略可以用来解决具体的数值目标。

example (h₀ : d ≤ e) : c + exp (a + d) ≤ c + exp (a + e) := by sorry

example : (0 : ℝ) < 1 := by norm_num

example (h : a ≤ b) : log (1 + exp a) ≤ log (1 + exp b) := by
  have h₀ : 0 < 1 + exp a := by sorry
  apply log_le_log h₀
  sorry

从这些例子应当可以看出，能够找到所需的库定理，是形式化工作的重要组成部分。你可以使用若干策略：

  • 可以在 Mathlib 的 GitHub 仓库中浏览 Mathlib。
  • 可以使用 Mathlib 网页上的 API 文档。
  • 可以使用 Loogle 通过模式搜索 Lean 与 Mathlib 中的定义和定理。
  • 可以依靠 Mathlib 的命名约定以及编辑器中的 Ctrl-space 补全（Mac 键盘上为 Cmd-space）来猜测定理名称。在 Lean 中，名为 A_of_B_of_C 的定理，通常从形如 B 和 C 的假设推出形如 A 的结论，其中 A、B 和 C 近似反映我们朗读目标的方式。因此，一个证明类似 x + y ≤ ... 的定理很可能以 add_le 开头。输入 add_le 并按 Ctrl-space，会出现一些有用的候选。注意，连续按两次 Ctrl-space 会显示关于可用补全的更多信息。
  • 如果在 VS Code 中右击一个已有定理名，编辑器会显示一个菜单，其中有跳转到该定理定义所在文件的选项；你也可以在附近找到类似定理。
  • 可以使用 apply? 策略，它会尝试在库中寻找相关定理。
example : 0 ≤ a ^ 2 := by
  -- apply?
  exact sq_nonneg a

若要在这个例子中尝试 apply?，请删除 exact 命令，并取消前一行的注释。借助这些技巧，看看能否找到完成下一个例子所需的内容：

example (h : a ≤ b) : c - exp b ≤ c - exp a := by
  sorry

使用同样的技巧，确认用 linarith 代替 apply? 也可以完成证明。

下面是另一个不等式的例子：

example : 2*a*b ≤ a^2 + b^2 := by
  have h : 0 ≤ a^2 - 2*a*b + b^2
  calc
    a^2 - 2*a*b + b^2 = (a - b)^2 := by ring
    _ ≥ 0 := by apply pow_two_nonneg

  calc
    2*a*b = 2*a*b + 0 := by ring
    _ ≤ 2*a*b + (a^2 - 2*a*b + b^2) := add_le_add (le_refl _) h
    _ = a^2 + b^2 := by ring

Mathlib 通常会在 * 和 ^ 这样的二元运算符两侧加入空格；但在这个例子中，更紧凑的格式反而提高了可读性。这里有几件事情值得注意。第一，表达式 s ≥ t 与 t ≤ s 定义相等。原则上，这意味着二者应当可以互换使用。但 Lean 的某些自动化不能识别这种等价，因此 Mathlib 往往偏好使用 ≤ 而不是 ≥。第二，我们大量使用了 ring 策略。它确实节省时间！最后，注意在第二个 calc 证明的第二行中，不必写 by exact add_le_add (le_refl _) h，可以直接写证明项 add_le_add (le_refl _) h。

事实上，上述证明中唯一需要一点技巧的地方，是找出假设 h。一旦有了它，第二个计算只涉及线性算术，linarith 可以处理：

example : 2*a*b ≤ a^2 + b^2 := by
  have h : 0 ≤ a^2 - 2*a*b + b^2
  calc
    a^2 - 2*a*b + b^2 = (a - b)^2 := by ring
    _ ≥ 0 := by apply pow_two_nonneg
  linarith

很好！我们邀请你用这些想法证明下面的定理。可以使用定理 abs_le'.mpr。还需要使用 constructor 策略把合取拆分成两个目标；参见 第 3.4 节。

example : |a*b| ≤ (a^2 + b^2)/2 := by
  sorry

#check abs_le'.mpr

如果你成功解决了它，恭喜！你已经很好地走在成为形式化大师的路上。

2.4. 使用 apply 和 rw 的更多例子

实数上的 min 函数由以下三个事实唯一刻画：

#check (min_le_left a b : min a b ≤ a)
#check (min_le_right a b : min a b ≤ b)
#check (le_min : c ≤ a → c ≤ b → c ≤ min a b)

你能猜出以类似方式刻画 max 的那些定理叫什么名字吗？

注意，我们必须通过写 min a b 而不是 min (a, b)，把 min 应用于一对参数 a 和 b。形式上，min 的类型是 ℝ → ℝ → ℝ。当我们写出带有多个箭头的类型时，约定是隐含括号向右结合，因此该类型解释为 ℝ → (ℝ → ℝ)。其效果是：若 a 和 b 的类型都是 ℝ，则 min a 的类型是 ℝ → ℝ，而 min a b 的类型是 ℝ，所以 min 如预期那样表现为一个二元函数。以这种方式处理多个参数称为柯里化，得名于逻辑学家 Haskell Curry。

Lean 中运算的优先级也需要一些时间适应。函数应用比中缀运算结合得更紧，因此表达式 min a b + c 会被解释为 (min a b) + c。随着练习增加，这些约定会逐渐变成自然习惯。

使用定理 le_antisymm，如果两个实数彼此小于等于对方，就可以证明它们相等。借助这一点和上述事实，我们可以证明 min 是交换的：

example : min a b = min b a := by
  apply le_antisymm
  · show min a b ≤ min b a
    apply le_min
    · apply min_le_right
    apply min_le_left
  · show min b a ≤ min a b
    apply le_min
    · apply min_le_right
    apply min_le_left

这里我们使用圆点来分隔不同目标的证明。我们的用法并不完全一致：在外层，我们对两个目标都使用圆点和缩进；而在嵌套证明中，只在还剩多个目标时使用圆点。两种约定都是合理且有用的。我们还使用 show 策略来组织证明，并指明每个代码块正在证明什么。即使没有这些 show 命令，证明仍然成立；但使用它们会使证明更易阅读和维护。

这个证明可能显得重复。为了预示稍后将学习的技巧，我们指出避免重复的一种方法是先陈述一个局部引理，然后使用它：

example : min a b = min b a := by
  have h : ∀ x y : ℝ, min x y ≤ min y x := by
    intro x y
    apply le_min
    apply min_le_right
    apply min_le_left
  apply le_antisymm
  apply h
  apply h

我们将在 第 3.1 节 中更多讨论全称量词；这里暂且只需知道，假设 h 表示对任意 x 和 y，所需不等式都成立，而 intro 策略引入任意的 x 和 y 来建立结论。le_antisymm 之后的第一个 apply 隐式使用 h a b，第二个则使用 h b a。

另一个解法是使用 repeat 策略；它会尽可能多次地应用某个策略（或策略块）。

example : min a b = min b a := by
  apply le_antisymm
  repeat
    apply le_min
    apply min_le_right
    apply min_le_left

我们鼓励你把下面这些命题作为练习来证明。第一个可以用刚才描述的任一技巧来缩短证明。

example : max a b = max b a := by
  sorry
example : min (min a b) c = min a (min b c) := by
  sorry

当然，你也可以证明 max 的结合律。

一个有趣的事实是，min 对 max 的分配方式类似于乘法对加法的分配，反之亦然。换言之，在实数上，我们有恒等式 min a (max b c) = max (min a b) (min a c)，也有把 max 与 min 对调后的对应版本。但下一节会看到，这些结论并不只由 ≤ 的传递性、自反性以及上面列出的 min 和 max 的刻画性质推出。我们还需要使用实数上的 ≤ 是全序这一事实，也就是说，它满足 ∀ x y, x ≤ y ∨ y ≤ x。这里的析取符号 ∨ 表示“或”。在第一种情形中有 min x y = x，在第二种情形中有 min x y = y。我们将在 第 3.5 节 中学习如何按情形推理；但目前我们只处理不需要拆分情形的例子。

下面就是这样的一个例子：

theorem aux : min a b + c ≤ min (a + c) (b + c) := by
  sorry
example : min a b + c = min (a + c) (b + c) := by
  sorry

显然，aux 提供了证明等式所需的两个不等式之一；但把它应用于合适的值，也能得到另一个方向。作为提示，可以使用定理 add_neg_cancel_right 和策略 linarith。

库中三角不等式的名称清楚体现了 Lean 的命名约定：

#check (abs_add_le : ∀ a b : ℝ, |a + b| ≤ |a| + |b|)

请使用它证明下面的变体，同时也使用 add_sub_cancel_right：

example : |a| - |b| ≤ |a - b| :=
  sorry
end

看看你能否在三行以内完成。可以使用定理 sub_add_cancel。

我们在后续各节还会使用的另一个重要关系，是自然数上的整除关系 x ∣ y。请注意，整除符号不是键盘上的普通竖线；它是一个 Unicode 字符，在 VS Code 中可通过输入 \| 获得。按约定，Mathlib 在定理名中用 dvd 表示它。

example (h₀ : x ∣ y) (h₁ : y ∣ z) : x ∣ z :=
  dvd_trans h₀ h₁

example : x ∣ y * x * z := by
  apply dvd_mul_of_dvd_left
  apply dvd_mul_left

example : x ∣ x ^ 2 := by
  apply dvd_mul_left

在最后一个例子中，指数是自然数，应用 dvd_mul_left 会迫使 Lean 把 x^2 的定义展开为 x^1 * x。看看你能否猜出证明下面命题所需的定理名称：

example (h : x ∣ w) : x ∣ y * (x * z) + x ^ 2 + w ^ 2 := by
  sorry
end

就整除关系而言，最大公因数 gcd 和最小公倍数 lcm 类似于 min 和 max。由于每个数都整除 0，因此 0 在整除意义下实际上是最大元素：

variable (m n : ℕ)

#check (Nat.gcd_zero_right n : Nat.gcd n 0 = n)
#check (Nat.gcd_zero_left n : Nat.gcd 0 n = n)
#check (Nat.lcm_zero_right n : Nat.lcm n 0 = 0)
#check (Nat.lcm_zero_left n : Nat.lcm 0 n = 0)

看看你能否猜出证明下面命题所需的定理名称：

example : Nat.gcd m n = Nat.gcd n m := by
  sorry

提示：可以使用 dvd_antisymm；但若这样做，Lean 会抱怨该表达式在一般定理和专门用于自然数的版本 Nat.dvd_antisymm 之间有歧义。可以用 _root_.dvd_antisymm 指明一般版本；两者都能工作。

2.5. 证明关于代数结构的事实

在 第 2.2 节 中，我们看到支配实数的许多常见恒等式，也适用于更一般的代数结构类，例如交换环。描述代数结构时，我们可以使用任意想要的公理，而不只限于等式。例如，一个偏序由一个集合及其上的一个二元关系组成，该关系满足自反性、传递性和反对称性，就像实数上的 ≤ 一样。Lean 知道偏序：

variable {α : Type*} [PartialOrder α]
variable (x y z : α)

#check x ≤ y
#check (le_refl x : x ≤ x)
#check (le_trans : x ≤ y → y ≤ z → x ≤ z)
#check (le_antisymm : x ≤ y → y ≤ x → x = y)

这里我们采用 Mathlib 的约定，用 α、β 和 γ（分别输入为 \a、\b 和 \g）表示任意类型。库中常用 R 和 G 这样的字母分别表示环、群等代数结构的载体；但一般来说，类型通常用希腊字母表示，尤其是在它们几乎没有附加结构时。

任一偏序 ≤ 还伴随一个严格偏序 <，其作用有些类似于实数上的 <。说 x 在该序下小于 y，等价于说它小于等于 y 且不等于 y。

#check x < y
#check (lt_irrefl x : ¬ (x < x))
#check (lt_trans : x < y → y < z → x < z)
#check (lt_of_le_of_lt : x ≤ y → y < z → x < z)
#check (lt_of_lt_of_le : x < y → y ≤ z → x < z)

example : x < y ↔ x ≤ y ∧ x ≠ y :=
  lt_iff_le_and_ne

在这个例子中，符号 ∧ 表示“且”，符号 ¬ 表示“非”，而 x ≠ y 是 ¬ (x = y) 的缩写。在 第 3 章 中，你将学习如何使用这些逻辑连接词证明 < 具有上述性质。

格是一种结构，它在偏序上增加两个运算 ⊓ 和 ⊔，分别类似于实数上的 min 和 max：

variable {α : Type*} [Lattice α]
variable (x y z : α)

#check x ⊓ y
#check (inf_le_left : x ⊓ y ≤ x)
#check (inf_le_right : x ⊓ y ≤ y)
#check (le_inf : z ≤ x → z ≤ y → z ≤ x ⊓ y)
#check x ⊔ y
#check (le_sup_left : x ≤ x ⊔ y)
#check (le_sup_right : y ≤ x ⊔ y)
#check (sup_le : x ≤ z → y ≤ z → x ⊔ y ≤ z)

⊓ 和 ⊔ 的刻画分别说明它们可称为最大下界和最小上界。在 VS Code 中，可以用 \glb 和 \lub 输入这些符号。它们也常称为下确界和上确界，Mathlib 在定理名中用 inf 和 sup 指称它们。更复杂的是，它们还常称为交和并。因此，如果处理格，就需要记住如下术语对应表：

  • ⊓ 是最大下界、下确界，或交。
  • ⊔ 是最小上界、上确界，或并。

格的一些实例包括：

  • 任意全序上的 min 和 max，例如整数或实数上的 ≤
  • 某个论域的子集族上的 ∩ 和 ∪，其序关系为 ⊆
  • 布尔真值上的 ∧ 和 ∨，其序关系 x ≤ y 表示 x 为假或 y 为真
  • 自然数（或正自然数）上的 gcd 和 lcm，其序关系是整除关系 ∣
  • 向量空间的线性子空间族，其中最大下界由交给出，最小上界由两个空间的和给出，序关系为包含关系
  • 一个集合（或在 Lean 中，一个类型）上的拓扑族，其中两个拓扑的最大下界是由它们的并生成的拓扑，最小上界是它们的交，序关系为反向包含

你可以验证，正如 min / max 以及 gcd / lcm 那样，仅凭下确界和上确界的刻画公理，再加上 le_refl 与 le_trans，即可证明它们的交换律和结合律。

当目标为 x ≤ z 时使用 apply le_trans 并不是好主意。实际上，Lean 无法猜出我们想使用哪个中间元素 y。因此，apply le_trans 会产生三个形如 x ≤ ?a、?a ≤ z 和 α 的目标，其中 ?a（可能有一个更复杂的自动生成名称）代表神秘的 y。最后一个类型为 α 的目标，是要求给出 y 的值。它排在最后，是因为 Lean 希望能从第一个目标 x ≤ ?a 的证明中自动推断它。为了避免这种不便，可以使用 calc 策略显式给出 y。或者，也可以使用 trans 策略；它以 y 为参数，并产生预期的目标 x ≤ y 和 y ≤ z。当然，也可以直接给出完整证明来避免这个问题，例如 exact le_trans inf_le_left inf_le_right，但这需要更多预先规划。

example : x ⊓ y = y ⊓ x := by
  sorry

example : x ⊓ y ⊓ z = x ⊓ (y ⊓ z) := by
  sorry

example : x ⊔ y = y ⊔ x := by
  sorry

example : x ⊔ y ⊔ z = x ⊔ (y ⊔ z) := by
  sorry

这些定理在 Mathlib 中分别名为 inf_comm、inf_assoc、sup_comm 和 sup_assoc。

另一个很好的练习是仅用这些公理证明吸收律：

theorem absorb1 : x ⊓ (x ⊔ y) = x := by
  sorry

theorem absorb2 : x ⊔ x ⊓ y = x := by
  sorry

这些定理在 Mathlib 中可分别以名称 inf_sup_self 和 sup_inf_self 找到。

若一个格还满足额外恒等式 x ⊓ (y ⊔ z) = (x ⊓ y) ⊔ (x ⊓ z) 和 x ⊔ (y ⊓ z) = (x ⊔ y) ⊓ (x ⊔ z)，则称为分配格。Lean 也知道这些结构：

variable {α : Type*} [DistribLattice α]
variable (x y z : α)

#check (inf_sup_left x y z : x ⊓ (y ⊔ z) = x ⊓ y ⊔ x ⊓ z)
#check (inf_sup_right x y z : (x ⊔ y) ⊓ z = x ⊓ z ⊔ y ⊓ z)
#check (sup_inf_left x y z : x ⊔ y ⊓ z = (x ⊔ y) ⊓ (x ⊔ z))
#check (sup_inf_right x y z : x ⊓ y ⊔ z = (x ⊔ z) ⊓ (y ⊔ z))

在 ⊓ 和 ⊔ 交换律成立的前提下，左右两个版本很容易证明等价。给出一个有限元素的非分配格的显式描述，从而说明并非每个格都是分配格，是一个很好的练习。另一个好练习是证明：在任意格中，任一条分配律都蕴含另一条分配律：

variable {α : Type*} [Lattice α]
variable (a b c : α)

example (h : ∀ x y z : α, x ⊓ (y ⊔ z) = x ⊓ y ⊔ x ⊓ z) : a ⊔ b ⊓ c = (a ⊔ b) ⊓ (a ⊔ c) := by
  sorry

example (h : ∀ x y z : α, x ⊔ y ⊓ z = (x ⊔ y) ⊓ (x ⊔ z)) : a ⊓ (b ⊔ c) = a ⊓ b ⊔ a ⊓ c := by
  sorry

可以把公理化结构组合成更大的结构。例如，严格有序环由一个环以及其载体上的一个偏序组成，并满足额外公理，表明环运算与该序相容：

variable {R : Type*} [Ring R] [PartialOrder R] [IsStrictOrderedRing R]
variable (a b c : R)

#check (add_le_add_right : a ≤ b → ∀ c, c + a ≤ c + b)
#check (mul_pos : 0 < a → 0 < b → 0 < a * b)

第 3 章 将提供从 mul_pos 和 < 的定义推出下列结论的方法：

#check (mul_nonneg : 0 ≤ a → 0 ≤ b → 0 ≤ a * b)

随后可以把它作为一个较长练习来证明：许多用于实数算术和实数序关系推理的常见事实，对任意有序环也一般成立。下面有几个例子可供尝试，只使用环、偏序的性质，以及最后两个例子中列出的事实（注意，这些环并未假定为交换环，因此不能使用 ring 策略）：

example (h : a ≤ b) : 0 ≤ b - a := by
  sorry

example (h: 0 ≤ b - a) : a ≤ b := by
  sorry

example (h : a ≤ b) (h' : 0 ≤ c) : a * c ≤ b * c := by
  sorry

最后，再看一个例子。度量空间由一个集合以及一个距离概念 dist x y 组成，后者把任意一对元素映射到一个实数。距离函数假定满足如下公理：

variable {X : Type*} [MetricSpace X]
variable (x y z : X)

#check (dist_self x : dist x x = 0)
#check (dist_comm x y : dist x y = dist y x)
#check (dist_triangle x y z : dist x z ≤ dist x y + dist y z)

掌握本节内容之后，你可以证明这些公理推出距离总是非负的：

example (x y : X) : 0 ≤ dist x y := by
  sorry

如你可能已经猜到的，这个定理在 Mathlib 中名为 dist_nonneg。
