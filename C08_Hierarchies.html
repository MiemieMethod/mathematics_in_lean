8. 层级结构

我们已在Chapter 7中看到如何定义群类、构造该类的实例，以及如何构造交换环类的实例。但这里当然存在一个层级：交换环特别地也是加法群。本章将研究如何构造这类层级。它们出现在数学的所有分支中，但本章将强调代数方面的例子。

在进一步讨论如何使用已有层级之前就讨论如何构造层级，或许显得有些过早。但使用层级也需要对其底层技术有所理解。因此你大概仍应阅读本章，不过第一次阅读时不必强求记住所有内容；可以先读后续章节，再回到这里重读一遍。

本章中，我们将重新定义 Mathlib 中出现的许多对象的较简化版本，因此会用下标区分我们的版本。例如，Ring₁ 将是我们版本的 Ring。由于我们将逐步解释形式化结构的更强方法，这些下标有时会超过 1。

8.1. 基础

在 Lean 所有层级的最底部，我们会遇到携带数据的类。下面的类记录给定类型 α 配备了一个称为 one 的特殊元素。在这个阶段，它完全没有任何性质。

class One₁ (α : Type) where
  /-- The element one -/
  one : α

由于本章将更频繁地使用类，我们需要更详细地理解 class 命令在做什么。首先，上面的 class 命令定义了一个结构 One₁，带参数 α : Type 和单一字段 one。它还把该结构标记为类，使得对某个类型 α 而言类型为 One₁ α 的参数，只要被标记为实例隐式参数，也就是出现在方括号中，就可以用实例解析过程推断出来。这两个效果也可以通过带 class 属性的 structure 命令实现，即写 @[class] structure。但 class 命令还会确保 One₁ α 在自身字段中作为实例隐式参数出现。比较如下：

#check One₁.one -- One₁.one {α : Type} [self : One₁ α] : α

@[class] structure One₂ (α : Type) where
  /-- The element one -/
  one : α

#check One₂.one

在第二个检查中，可以看到 self : One₂ α 是显式参数。让我们确认第一个版本确实无需任何显式参数即可使用。

example (α : Type) [One₁ α] : α := One₁.one

备注：在上面的例子中，参数 One₁ α 被标记为实例隐式参数，这有点小题大做，因为它只影响声明的uses，而由 example 命令创建的声明无法被使用。不过，这使我们可以避免给该参数命名；更重要的是，它帮助我们养成把 One₁ α 参数标记为实例隐式参数的好习惯。

另一个备注是，只有当 Lean 知道 α 是什么时，这一切才会工作。在上面的例子中，如果省略类型标注 : α，会生成类似 typeclass instance problem is stuck, it is often due to metavariables One₁ (?m.263 α) 的错误信息，其中 ?m.263 α 表示“某个依赖于 α 的类型”（263 只是自动生成的下标，用于区分多个未知对象）。避免此问题的另一种方法是使用类型标注，例如：

example (α : Type) [One₁ α] := (One₁.one : α)

你在Section 3.6中尝试序列极限时，或许已经遇到过这个问题；例如，若想陈述 0 < 1，却没有告诉 Lean 这个不等式是在自然数中还是实数中。

下一项任务是给 One₁.one 指派一个记号。由于不希望与内置记号 1 冲突，我们将使用 𝟙。下面的命令实现了这一点，其中第一行告诉 Lean 使用 One₁.one 的文档作为符号 𝟙 的文档。

@[inherit_doc]
notation "𝟙" => One₁.one

example {α : Type} [One₁ α] : α := 𝟙

example {α : Type} [One₁ α] : (𝟙 : α) = 𝟙 := rfl

现在我们想要一个携带数据的类，用来记录一个二元运算。暂时不想在加法和乘法之间作选择，所以将使用菱形符号。

class Dia₁ (α : Type) where
  dia : α → α → α

infixl:70 " ⋄ "   => Dia₁.dia

与 One₁ 的例子一样，此阶段该运算完全没有任何性质。现在定义半群结构类，其中运算记作 ⋄。暂时先手工把它定义为一个带两个字段的结构：一个 Dia₁ 实例，以及一个取值于 Prop 的字段 dia_assoc，断言 ⋄ 满足结合律。

class Semigroup₀ (α : Type) where
  toDia₁ : Dia₁ α
  /-- Diamond is associative -/
  dia_assoc : ∀ a b c : α, a ⋄ b ⋄ c = a ⋄ (b ⋄ c)

注意，在陈述 dia_assoc 时，先前定义的字段 toDia₁ 位于局部上下文中，因此 Lean 在寻找 Dia₁ α 的实例以理解 a ⋄ b 时可以使用它。然而这个 toDia₁ 字段不会成为类型类实例数据库的一部分。因此，执行 example {α : Type} [Semigroup₁ α] (a b : α) : α := a ⋄ b 会失败，并给出错误信息 failed to synthesize instance Dia₁ α。

可以稍后添加 instance 属性来修正这一点。

attribute [instance] Semigroup₀.toDia₁

example {α : Type} [Semigroup₀ α] (a b : α) : α := a ⋄ b

在继续向上构造之前，需要用一种不同语法加入这个 toDia₁ 字段，告诉 Lean 应把 Dia₁ α 当作其字段就是 Semigroup₁ 自身的字段来处理。这也会方便地自动添加 toDia₁ 实例。class 命令通过 extends 语法支持这一点，如下所示：

class Semigroup₁ (α : Type) extends toDia₁ : Dia₁ α where
  /-- Diamond is associative -/
  dia_assoc : ∀ a b c : α, a ⋄ b ⋄ c = a ⋄ (b ⋄ c)

example {α : Type} [Semigroup₁ α] (a b : α) : α := a ⋄ b

注意，这个语法也可用于 structure 命令；不过在那种情况下，由于没有要定义的实例，它只解决书写诸如 toDia₁ 这类字段的麻烦。

在 extends 语法中，字段名 toDia₁ 是可选的。默认情况下，它取被扩展类的名称，并在前面加上“to”。

class Semigroup₂ (α : Type) extends Dia₁ α where
  /-- Diamond is associative -/
  dia_assoc : ∀ a b c : α, a ⋄ b ⋄ c = a ⋄ (b ⋄ c)

现在尝试把一个菱形运算和一个特殊的一元素结合起来，并加入公理说明该元素在左右两边都是单位元。

class DiaOneClass₁ (α : Type) extends One₁ α, Dia₁ α where
  /-- One is a left neutral element for diamond. -/
  one_dia : ∀ a : α, 𝟙 ⋄ a = a
  /-- One is a right neutral element for diamond -/
  dia_one : ∀ a : α, a ⋄ 𝟙 = a

在下一个例子中，我们告诉 Lean：α 具有 DiaOneClass₁ 结构，并陈述一个同时使用 Dia₁ 实例和 One₁ 实例的性质。为了观察 Lean 如何找到这些实例，我们设置一个跟踪选项，其结果可在 Infoview 中看到。默认结果相当简略，但可以点击以黑色箭头结尾的行展开。它包含一些失败尝试，即 Lean 在尚未拥有足够类型信息前试图寻找实例。成功的尝试确实会用到由 extends 语法生成的实例。

set_option trace.Meta.synthInstance true in
example {α : Type} [DiaOneClass₁ α] (a b : α) : Prop := a ⋄ b = 𝟙

注意，在组合已有类时，不需要加入额外字段。因此可以如下定义幺半群：

class Monoid₁ (α : Type) extends Semigroup₁ α, DiaOneClass₁ α

虽然上面的定义看起来直接，但它隐藏了一个重要细节。Semigroup₁ α 和 DiaOneClass₁ α 都扩展了 Dia₁ α，因此人们可能担心，拥有一个 Monoid₁ α 实例会给出 α 上两个无关的菱形运算：一个来自字段 Monoid₁.toSemigroup₁，另一个来自字段 Monoid₁.toDiaOneClass₁。

确实，如果尝试手工构造一个幺半群类：

class Monoid₂ (α : Type) where
  toSemigroup₁ : Semigroup₁ α
  toDiaOneClass₁ : DiaOneClass₁ α

那么会得到两个完全无关的菱形运算 Monoid₂.toSemigroup₁.toDia₁.dia 和 Monoid₂.toDiaOneClass₁.toDia₁.dia。

使用 extends 语法生成的版本没有这个缺陷。

example {α : Type} [Monoid₁ α] :
  (Monoid₁.toSemigroup₁.toDia₁.dia : α → α → α) = Monoid₁.toDiaOneClass₁.toDia₁.dia := rfl

所以 class 命令为我们做了一些魔法（structure 命令也会如此）。查看这些类的字段的一种简单方法，是检查它们的构造子。比较如下：

/- Monoid₂.mk {α : Type} (toSemigroup₁ : Semigroup₁ α) (toDiaOneClass₁ : DiaOneClass₁ α) : Monoid₂ α -/
#check Monoid₂.mk

/- Monoid₁.mk {α : Type} [toSemigroup₁ : Semigroup₁ α] [toOne₁ : One₁ α] (one_dia : ∀ (a : α), 𝟙 ⋄ a = a) (dia_one : ∀ (a : α), a ⋄ 𝟙 = a) : Monoid₁ α -/
#check Monoid₁.mk

于是可见，Monoid₁ 如预期接受 Semigroup₁ α 参数，但不会接受一个可能重叠的 DiaOneClass₁ α 参数，而是把它拆开，只包含不重叠的部分。它还自动生成了实例 Monoid₁.toDiaOneClass₁；该实例not字段，但具有预期的签名。从最终用户角度看，这恢复了两个被扩展类 Semigroup₁ 和 DiaOneClass₁ 之间的对称性。

#check Monoid₁.toSemigroup₁
#check Monoid₁.toDiaOneClass₁

现在我们已经非常接近定义群。可以向幺半群结构添加一个字段，断言每个元素都存在逆元。但这样之后还需要努力访问这些逆元。实践中，把逆元作为数据加入更方便。为了优化可复用性，我们定义一个新的携带数据的类，然后给它一些记号。

class Inv₁ (α : Type) where
  /-- The inversion function -/
  inv : α → α

@[inherit_doc]
postfix:max "⁻¹" => Inv₁.inv

class Group₁ (G : Type) extends Monoid₁ G, Inv₁ G where
  inv_dia : ∀ a : G, a⁻¹ ⋄ a = 𝟙

上面的定义可能看起来太弱，因为只要求 a⁻¹ 是 a 的左逆。但另一边会自动成立。为了证明这一点，需要一个预备引理。

lemma left_inv_eq_right_inv₁ {M : Type} [Monoid₁ M] {a b c : M} (hba : b ⋄ a = 𝟙) (hac : a ⋄ c = 𝟙) : b = c := by
  rw [← DiaOneClass₁.one_dia c, ← hba, Semigroup₁.dia_assoc, hac, DiaOneClass₁.dia_one b]

在这个引理中，给出全名相当烦人，尤其是它要求知道层级的哪一部分提供了这些事实。一种修正方法是使用 export 命令，把这些事实复制成根命名空间中的引理。

export DiaOneClass₁ (one_dia dia_one)
export Semigroup₁ (dia_assoc)
export Group₁ (inv_dia)

于是可以把上面的证明重写为：

example {M : Type} [Monoid₁ M] {a b c : M} (hba : b ⋄ a = 𝟙) (hac : a ⋄ c = 𝟙) : b = c := by
  rw [← one_dia c, ← hba, dia_assoc, hac, dia_one b]

现在轮到你证明关于我们这些代数结构的事实了。

lemma inv_eq_of_dia [Group₁ G] {a b : G} (h : a ⋄ b = 𝟙) : a⁻¹ = b :=
  sorry

lemma dia_inv [Group₁ G] (a : G) : a ⋄ a⁻¹ = 𝟙 :=
  sorry

在这个阶段，我们想继续定义环，但这里有一个严重问题。一个类型上的环结构同时包含加法群结构、乘法幺半群结构，以及关于二者相互作用的一些性质。但到目前为止，我们为所有运算都硬编码了记号 ⋄。更根本地，类型类系统假定每个类型对每个类型类只有一个实例。有多种方式解决这个问题。令人意外的是，Mathlib 借助某个生成代码的属性，采用朴素思路：为加法理论和乘法理论各复制一份。结构和类都会以加法记号和乘法记号定义，并用属性 to_additive 关联起来。对于半群这类多重继承情形，自动生成的“恢复对称性”实例也需要被标记。这有些技术性，不必理解细节。关键点是：引理随后只用乘法记号陈述，并标记属性 to_additive，以生成加法版本；例如 left_inv_eq_right_inv' 会自动生成加法版本 left_neg_eq_right_neg'。为了检查这个加法版本的名称，我们在 left_inv_eq_right_inv' 上方使用了命令 whatsnew in。

class AddSemigroup₃ (α : Type) extends Add α where
  /-- Addition is associative -/
  add_assoc₃ : ∀ a b c : α, a + b + c = a + (b + c)

@[to_additive AddSemigroup₃]
class Semigroup₃ (α : Type) extends Mul α where
  /-- Multiplication is associative -/
  mul_assoc₃ : ∀ a b c : α, a * b * c = a * (b * c)

class AddMonoid₃ (α : Type) extends AddSemigroup₃ α, AddZeroClass α

@[to_additive AddMonoid₃]
class Monoid₃ (α : Type) extends Semigroup₃ α, MulOneClass α

export Semigroup₃ (mul_assoc₃)
export AddSemigroup₃ (add_assoc₃)

whatsnew in
@[to_additive]
lemma left_inv_eq_right_inv' {M : Type} [Monoid₃ M] {a b c : M} (hba : b * a = 1) (hac : a * c = 1) : b = c := by
  rw [← one_mul c, ← hba, mul_assoc₃, hac, mul_one b]

#check left_neg_eq_right_neg'

有了这项技术，我们也可以轻松定义交换半群、交换幺半群和交换群，然后定义环。

class AddCommSemigroup₃ (α : Type) extends AddSemigroup₃ α where
  add_comm : ∀ a b : α, a + b = b + a

@[to_additive AddCommSemigroup₃]
class CommSemigroup₃ (α : Type) extends Semigroup₃ α where
  mul_comm : ∀ a b : α, a * b = b * a

class AddCommMonoid₃ (α : Type) extends AddMonoid₃ α, AddCommSemigroup₃ α

@[to_additive AddCommMonoid₃]
class CommMonoid₃ (α : Type) extends Monoid₃ α, CommSemigroup₃ α

class AddGroup₃ (G : Type) extends AddMonoid₃ G, Neg G where
  neg_add : ∀ a : G, -a + a = 0

@[to_additive AddGroup₃]
class Group₃ (G : Type) extends Monoid₃ G, Inv G where
  inv_mul : ∀ a : G, a⁻¹ * a = 1

现在可以定义环了。为了演示，我们不假设加法可交换，然后立即提供一个 AddCommGroup₃ 实例。Mathlib 并不这样做，首先因为实践中这不会让任何环实例更容易；其次是因为 Mathlib 的代数层级经过半环，而半环类似于环但没有相反元，所以下面的证明对它们不适用。这里的收获，除了给从未见过此事的读者一个漂亮练习外，是展示如何使用一种语法构造实例：它允许把父结构作为实例参数提供，然后补充额外字段。这里参数 Ring₃ R 提供了 AddCommGroup₃ R 所需的除 add_comm 之外的一切。

class Ring₃ (R : Type) extends AddGroup₃ R, Monoid₃ R, MulZeroClass R where
  /-- Multiplication is left distributive over addition -/
  left_distrib : ∀ a b c : R, a * (b + c) = a * b + a * c
  /-- Multiplication is right distributive over addition -/
  right_distrib : ∀ a b c : R, (a + b) * c = a * c + b * c

instance {R : Type} [Ring₃ R] : AddCommGroup₃ R :=
{ add_comm := by
    sorry }

当然，也可以构造具体实例，例如整数上的环结构（当然，下面的实例使用了 Mathlib 已完成的全部工作）。

instance : Ring₃ ℤ where
  add := (· + ·)
  add_assoc₃ := add_assoc
  zero := 0
  zero_add := by simp
  add_zero := by simp
  neg := (- ·)
  neg_add := by simp
  mul := (· * ·)
  mul_assoc₃ := mul_assoc
  one := 1
  one_mul := by simp
  mul_one := by simp
  zero_mul := by simp
  mul_zero := by simp
  left_distrib := Int.mul_add
  right_distrib := Int.add_mul

作为练习，你现在可以为序关系建立一个简单层级，其中包括有序交换幺半群的类；这类结构同时具有偏序和交换幺半群结构，并满足 ∀ a b : α, a ≤ b → ∀ c : α, c * a ≤ c * b。当然，你需要给下面这些类添加字段，也许还要添加 extends 子句。

class LE₁ (α : Type) where
  /-- The Less-or-Equal relation. -/
  le : α → α → Prop

@[inherit_doc] infix:50 " ≤₁ " => LE₁.le

class Preorder₁ (α : Type)

class PartialOrder₁ (α : Type)

class OrderedCommMonoid₁ (α : Type)

现在讨论涉及多个类型的代数结构。首要例子是环上的模。如果你不知道什么是模，可以假装它表示向量空间，并认为所有环都是域。这些结构是交换加法群，并配备来自某个环的元素的标量乘法。

首先定义一个携带数据的类型类，记录某个类型 α 对某个类型 β 的标量乘法，并给它一个右结合记号。

class SMul₃ (α : Type) (β : Type) where
  /-- Scalar multiplication -/
  smul : α → β → β

infixr:73 " • " => SMul₃.smul

然后可以定义模（如果不知道什么是模，仍可把它想成向量空间）。

class Module₁ (R : Type) [Ring₃ R] (M : Type) [AddCommGroup₃ M] extends SMul₃ R M where
  zero_smul : ∀ m : M, (0 : R) • m = 0
  one_smul : ∀ m : M, (1 : R) • m = m
  mul_smul : ∀ (a b : R) (m : M), (a * b) • m = a • b • m
  add_smul : ∀ (a b : R) (m : M), (a + b) • m = a • m + b • m
  smul_add : ∀ (a : R) (m n : M), a • (m + n) = a • m + a • n

这里发生了一件有趣的事。环 R 上的环结构是这个定义的参数，这并不太令人惊讶；但你可能预期 AddCommGroup₃ M 会像 SMul₃ R M 一样成为 extends 子句的一部分。尝试这么做会产生一个标记为实例的字段 Module₃.toAddCommGroup₃。这个实例会有签名 (R : Type) → [inst : Ring₃ R] → {M : Type} → [self : Module₁ R M] → AddCommGroup₃ M。有了这样的实例在类型类数据库中，每当 Lean 为某个 M 寻找 AddCommGroup₃ M 实例时，它都必须在开始寻找 Module₁ R M 实例这一主任务之前，先追寻完全未指定的类型 R 和 Ring₃ R 实例。这两个支线任务由错误信息中提到的元变量表示，那里记为 ?R 和 ?inst✝。这样的 Module₃.toAddCommGroup₃ 实例会成为实例解析过程的巨大陷阱。

那么 extends SMul₃ R M 呢？它会创建字段 Module₁.toSMul₃ : {R : Type} →  [inst : Ring₃ R] → {M : Type} → [inst_1 : AddCommGroup₃ M] → [self : Module₁ R M] → SMul₃ R M，其最终结果 SMul₃ R M 同时提到 R 和 M，因此这个字段可以安全地作为实例使用。规则很容易记：extends 子句中出现的每个类都应提到参数中出现的每个类型。

让我们创建第一个模实例：一个环以其乘法作为标量乘法，是自身上的模。

instance selfModule (R : Type) [Ring₃ R] : Module₁ R R where
  smul := fun r s ↦ r*s
  zero_smul := zero_mul
  one_smul := one_mul
  mul_smul := mul_assoc₃
  add_smul := Ring₃.right_distrib
  smul_add := Ring₃.left_distrib

作为第二个例子，每个阿贝尔群都是 ℤ 上的模（这是允许不可逆标量以推广向量空间理论的原因之一）。首先，对于任意配备零元和加法的类型，可以定义自然数标量乘法：n • a 定义为 a + ⋯ + a，其中 a 出现 n 次。然后通过确保 (-1) • a = -a，把它扩展为整数标量乘法。

def nsmul₁ {M : Type*} [Zero M] [Add M] : ℕ → M → M
  | 0, _ => 0
  | n + 1, a => a + nsmul₁ n a

def zsmul₁ {M : Type*} [Zero M] [Add M] [Neg M] : ℤ → M → M
  | Int.ofNat n, a => nsmul₁ n a
  | Int.negSucc n, a => -nsmul₁ n.succ a

证明这确实给出模结构有些冗长，而且与当前讨论无关，因此我们会用 sorry 填充所有公理。并not要求你把这些 sorry 替换为证明。如果坚持要做，大概需要陈述并证明若干关于 nsmul₁ 和 zsmul₁ 的中间引理。

instance abGrpModule (A : Type) [AddCommGroup₃ A] : Module₁ ℤ A where
  smul := zsmul₁
  zero_smul := sorry
  one_smul := sorry
  mul_smul := sorry
  add_smul := sorry
  smul_add := sorry

一个更重要的问题是，现在 ℤ 自身在环 ℤ 上有两个模结构：abGrpModule ℤ，因为 ℤ 是阿贝尔群；以及 selfModule ℤ，因为 ℤ 是环。这两个模结构对应同一个阿贝尔群结构，但它们是否有相同的标量乘法并不显然。事实上它们确实相同，但这不是按定义成立，而需要证明。对类型类实例解析过程而言，这是非常糟糕的消息，会给该层级的使用者带来令人沮丧的失败。当直接要求寻找一个实例时，Lean 会选取其中一个；我们可以用如下方式查看它选了哪个：

#synth Module₁ ℤ ℤ -- abGrpModule ℤ

但在更间接的语境中，Lean 可能推断出另一个实例，随后陷入混乱。这种情形称为坏菱形。这与前面使用的菱形运算无关，而是指可以把从 ℤ 到其 Module₁ ℤ 的路径画成经过 AddCommGroup₃ ℤ 或经过 Ring₃ ℤ 两条路。

需要理解，并非所有菱形都是坏的。事实上，Mathlib 中到处都有菱形，本章也是如此。早在开头我们就看到，可以从 Monoid₁ α 经过 Semigroup₁ α 或 DiaOneClass₁ α 到达 Dia₁ α；由于 class 命令所做的工作，得到的两个 Dia₁ α 实例按定义相等。特别地，若菱形底部是取值于 Prop 的类，它不可能是坏的，因为同一命题的任意两个证明按定义相等。

但我们用模创建出的菱形确实是坏的。出问题的部分是字段 smul，它是数据而不是证明，而且两种构造按定义并不相等。稳健的修正方式，是确保从丰富结构到贫乏结构总是通过忘却数据完成，而不是通过定义数据完成。这个著名模式称为“遗忘继承”，并在 https://inria.hal.science/hal-02463336v2 中得到广泛讨论。

在我们的具体情形中，可以修改 AddMonoid₃ 的定义，加入一个 nsmul 数据字段，以及若干取值于 Prop 的字段，以保证此操作可证地等于上面构造的操作。下面定义中，这些字段在类型之后用 := 给出默认值。由于这些默认值，大多数实例仍可与先前定义完全一样地构造。但在 ℤ 的特殊情形中，我们将能提供特定取值。

class AddMonoid₄ (M : Type) extends AddSemigroup₃ M, AddZeroClass M where
  /-- Multiplication by a natural number. -/
  nsmul : ℕ → M → M := nsmul₁
  /-- Multiplication by `(0 : ℕ)` gives `0`. -/
  nsmul_zero : ∀ x, nsmul 0 x = 0 := by intros; rfl
  /-- Multiplication by `(n + 1 : ℕ)` behaves as expected. -/
  nsmul_succ : ∀ (n : ℕ) (x), nsmul (n + 1) x = x + nsmul n x := by intros; rfl

instance mySMul {M : Type} [AddMonoid₄ M] : SMul ℕ M := ⟨AddMonoid₄.nsmul⟩

让我们检查，不提供与 nsmul 相关的字段时，仍能构造乘积幺半群实例。

instance (M N : Type) [AddMonoid₄ M] [AddMonoid₄ N] : AddMonoid₄ (M × N) where
  add := fun p q ↦ (p.1 + q.1, p.2 + q.2)
  add_assoc₃ := fun a b c ↦ by ext <;> apply add_assoc₃
  zero := (0, 0)
  zero_add := fun a ↦ by ext <;> apply zero_add
  add_zero := fun a ↦ by ext <;> apply add_zero

现在处理 ℤ 的特殊情况：希望使用从 ℕ 到 ℤ 的强制转换以及 ℤ 上的乘法来构造 nsmul。特别注意，这里的证明字段比上面默认值中的证明字段包含更多工作。

instance : AddMonoid₄ ℤ where
  add := (· + ·)
  add_assoc₃ := Int.add_assoc
  zero := 0
  zero_add := Int.zero_add
  add_zero := Int.add_zero
  nsmul := fun n m ↦ (n : ℤ) * m
  nsmul_zero := Int.zero_mul
  nsmul_succ := fun n m ↦ show (n + 1 : ℤ) * m = m + n * m
    by rw [Int.add_mul, Int.add_comm, Int.one_mul]

让我们检查问题是否已经解决。因为 Lean 已经有自然数和整数之间标量乘法的定义，而我们希望确保使用的是自己的实例，所以不使用记号 •，而是调用 SMul.mul 并显式提供上面定义的实例。

example (n : ℕ) (m : ℤ) : SMul.smul (self := mySMul) n m = n * m := rfl

这个故事随后会继续，把 zsmul 字段纳入群的定义，并采用类似技巧。现在你已经可以阅读 Mathlib 中幺半群、群、环和模的定义了。它们比这里看到的更复杂，因为它们是庞大层级的一部分，但所有原则都已在上文解释。

作为练习，可以回到上面构造的序关系层级，尝试纳入一个携带小于记号 <₁ 的类型类 LT₁，并确保每个预序都带有一个 <₁，其默认值由 ≤₁ 构造，并带有一个取值于 Prop 的字段，断言这两个比较运算之间的自然关系。

8.2. 态射

到目前为止，本章讨论了如何创建数学结构的层级。但在拥有态射之前，结构的定义还不算真正完成。这里有两种主要方法。最明显的一种是在函数上定义谓词。

def isMonoidHom₁ [Monoid G] [Monoid H] (f : G → H) : Prop :=
  f 1 = 1 ∧ ∀ g g', f (g * g') = f g * f g'

在这个定义中，使用合取略显不便。特别地，用户在想访问两个条件时，需要记住我们选择的顺序。因此可以改用一个结构。

structure isMonoidHom₂ [Monoid G] [Monoid H] (f : G → H) : Prop where
  map_one : f 1 = 1
  map_mul : ∀ g g', f (g * g') = f g * f g'

走到这一步，甚至很容易想把它做成一个类，并使用类型类实例解析过程，从较简单函数的实例自动推断复杂函数的 isMonoidHom₂。例如，幺半群态射的复合仍是幺半群态射，这似乎是一个有用实例。然而，这样的实例会使解析过程非常棘手，因为它需要到处搜寻 g ∘ f。若在 g (f x) 中找不到，会令人十分沮丧。更一般地说，必须始终记住，识别给定表达式中应用了哪个函数是一个非常困难的问题，称为“高阶合一问题”。因此 Mathlib 不采用这种类方法。

一个更根本的问题是：我们是像上面那样使用谓词（通过 def 或 structure），还是使用把函数和谓词打包在一起的结构。这部分也是心理层面的问题。考虑一个不是态射的幺半群之间函数极为少见。直觉上，“幺半群态射”并不像是可以赋给裸函数的形容词，而更像是一个名词。另一方面，也可以说拓扑空间之间的连续函数本质上是一个恰好连续的函数。这是 Mathlib 拥有谓词 Continuous 的原因之一。例如可以写：

example : Continuous (id : ℝ → ℝ) := continuous_id

我们仍有连续函数的打包版本，例如它们便于给连续函数空间赋予拓扑；但处理连续性时，它们不是主要工具。

相比之下，幺半群（或其他代数结构）之间的态射则按如下方式打包：

@[ext]
structure MonoidHom₁ (G H : Type) [Monoid G] [Monoid H]  where
  toFun : G → H
  map_one : toFun 1 = 1
  map_mul : ∀ g g', toFun (g * g') = toFun g * toFun g'

当然，我们不希望到处输入 toFun，所以使用 CoeFun 类型类注册一个强制转换。它的第一个参数是我们想强制转换为函数的类型。第二个参数描述目标函数类型。在我们的情形中，对每个 f : MonoidHom₁ G H，它总是 G → H。我们还用 coe 属性标记 MonoidHom₁.toFun，以确保它在策略状态中几乎不可见地显示，只带一个 ↑ 前缀。

instance [Monoid G] [Monoid H] : CoeFun (MonoidHom₁ G H) (fun _ ↦ G → H) where
  coe := MonoidHom₁.toFun

attribute [coe] MonoidHom₁.toFun

让我们检查，确实可以把打包的幺半群态射应用到一个元素上。

example [Monoid G] [Monoid H] (f : MonoidHom₁ G H) : f 1 = 1 :=  f.map_one

对其他类型的态射也可以这样做，直到得到环同态。

@[ext]
structure AddMonoidHom₁ (G H : Type) [AddMonoid G] [AddMonoid H]  where
  toFun : G → H
  map_zero : toFun 0 = 0
  map_add : ∀ g g', toFun (g + g') = toFun g + toFun g'

instance [AddMonoid G] [AddMonoid H] : CoeFun (AddMonoidHom₁ G H) (fun _ ↦ G → H) where
  coe := AddMonoidHom₁.toFun

attribute [coe] AddMonoidHom₁.toFun

@[ext]
structure RingHom₁ (R S : Type) [Ring R] [Ring S] extends MonoidHom₁ R S, AddMonoidHom₁ R S

这种方法有几个问题。一个小问题是，我们不太知道应把 coe 属性放在哪里，因为 RingHom₁.toFun 并不存在，相关函数是 MonoidHom₁.toFun ∘ RingHom₁.toMonoidHom₁，而它不是一个可以标记属性的声明（不过仍可定义一个 CoeFun  (RingHom₁ R S) (fun _ ↦ R → S) 实例）。一个重要得多的问题是，关于幺半群态射的引理不会直接应用到环同态上。这样一来，要么每次想应用幺半群态射引理时都要摆弄 RingHom₁.toMonoidHom₁，要么为环同态重述每个这类引理。两种选择都不理想，所以 Mathlib 在这里使用一个新的层级技巧。思路是为“至少是幺半群态射”的对象定义一个类型类，让幺半群态射和环同态都成为该类的实例，并用它来陈述每个引理。在下面的定义中，F 可以是 MonoidHom₁ M N；如果 M 和 N 有环结构，也可以是 RingHom₁ M N。

class MonoidHomClass₁ (F : Type) (M N : Type) [Monoid M] [Monoid N] where
  toFun : F → M → N
  map_one : ∀ f : F, toFun f 1 = 1
  map_mul : ∀ f g g', toFun f (g * g') = toFun f g * toFun f g'

不过，上面的实现还有一个问题。我们尚未注册到函数的强制转换实例。现在试着这样做。

def badInst [Monoid M] [Monoid N] [MonoidHomClass₁ F M N] : CoeFun F (fun _ ↦ M → N) where
  coe := MonoidHomClass₁.toFun

把它做成实例会很糟糕。当面对类似 f x 的表达式且 f 的类型不是函数类型时，Lean 会尝试寻找 CoeFun 实例，把 f 强制转换为函数。上面的函数具有类型 {M N F : Type} → [Monoid M] → [Monoid N] → [MonoidHomClass₁ F M N] → CoeFun F (fun x ↦ M → N)，因此在尝试应用它时，Lean 事先并不清楚未知类型 M、N 和 F 应按什么顺序推断。这是一种与前面见过的坏实例略有不同的坏实例，但归根到底是同一个问题：在不知道 M 的情况下，Lean 将不得不在未知类型上搜索幺半群实例，从而无望地尝试数据库中的every幺半群实例。如果好奇这种实例会产生什么效果，可以在上述声明上方输入 set_option synthInstance.checkSynthOrder false in，把 def badInst 替换成 instance，然后在本文件中寻找随机失败。

这里的解法很简单：需要告诉 Lean 先搜索 F 是什么，然后再推出 M 和 N。这通过函数 outParam 完成。该函数被定义为恒等函数，但类型类机制仍会识别它，并触发所需行为。因此可以注意函数 outParam，重新尝试定义我们的类：

class MonoidHomClass₂ (F : Type) (M N : outParam Type) [Monoid M] [Monoid N] where
  toFun : F → M → N
  map_one : ∀ f : F, toFun f 1 = 1
  map_mul : ∀ f g g', toFun f (g * g') = toFun f g * toFun f g'

instance [Monoid M] [Monoid N] [MonoidHomClass₂ F M N] : CoeFun F (fun _ ↦ M → N) where
  coe := MonoidHomClass₂.toFun

attribute [coe] MonoidHomClass₂.toFun

现在可以继续执行实例化这个类的计划。

instance (M N : Type) [Monoid M] [Monoid N] : MonoidHomClass₂ (MonoidHom₁ M N) M N where
  toFun := MonoidHom₁.toFun
  map_one := fun f ↦ f.map_one
  map_mul := fun f ↦ f.map_mul

instance (R S : Type) [Ring R] [Ring S] : MonoidHomClass₂ (RingHom₁ R S) R S where
  toFun := fun f ↦ f.toMonoidHom₁.toFun
  map_one := fun f ↦ f.toMonoidHom₁.map_one
  map_mul := fun f ↦ f.toMonoidHom₁.map_mul

如所承诺的，假设有 MonoidHomClass₁ F 的实例而对 f : F 证明的每个引理，都将同时适用于幺半群态射和环同态。来看一个示例引理，并检查它适用于两种情形。

lemma map_inv_of_inv [Monoid M] [Monoid N] [MonoidHomClass₂ F M N] (f : F) {m m' : M} (h : m*m' = 1) :
    f m * f m' = 1 := by
  rw [← MonoidHomClass₂.map_mul, h, MonoidHomClass₂.map_one]

example [Monoid M] [Monoid N] (f : MonoidHom₁ M N) {m m' : M} (h : m*m' = 1) : f m * f m' = 1 :=
map_inv_of_inv f h

example [Ring R] [Ring S] (f : RingHom₁ R S) {r r' : R} (h : r*r' = 1) : f r * f r' = 1 :=
map_inv_of_inv f h

乍看之下，这似乎又回到了把 MonoidHom₁ 做成类这一旧坏主意。但事实并非如此。一切都上移了一个抽象层级。类型类解析过程不会搜索函数，而会搜索 MonoidHom₁ 或 RingHom₁。

我们的方法还有一个遗留问题：围绕字段 toFun、相应的 CoeFun 实例和 coe 属性存在重复代码。更好的做法还应记录这一模式只用于带有额外性质的函数，这意味着到函数的强制转换应当是单射。因此 Mathlib 通过基类 DFunLike 又增加了一层抽象（其中 “DFun” 表示依赖函数）。让我们在这个基础层之上重新定义 MonoidHomClass。

class MonoidHomClass₃ (F : Type) (M N : outParam Type) [Monoid M] [Monoid N] extends
    DFunLike F M (fun _ ↦ N) where
  map_one : ∀ f : F, f 1 = 1
  map_mul : ∀ (f : F) g g', f (g * g') = f g * f g'

instance (M N : Type) [Monoid M] [Monoid N] : MonoidHomClass₃ (MonoidHom₁ M N) M N where
  coe := MonoidHom₁.toFun
  coe_injective' _ _ := MonoidHom₁.ext
  map_one := MonoidHom₁.map_one
  map_mul := MonoidHom₁.map_mul

当然，态射层级并不止于此。我们可以继续定义一个扩展 MonoidHomClass₃ 的类 RingHomClass₃，并在 RingHom 上实例化它，之后还可在 AlgebraHom 上实例化它（代数是在环上添加一些额外结构得到的）。不过，我们已经覆盖了 Mathlib 中用于态射的主要形式化思想，你应当已经准备好理解 Mathlib 中态射是如何定义的。

作为练习，你应尝试定义有序类型之间打包的保序函数类，然后定义保序幺半群态射。这只是训练用途。与连续函数一样，在 Mathlib 中保序函数主要以未打包形式使用，并由谓词 Monotone 定义。当然，你需要补全下面的类定义。

@[ext]
structure OrderPresHom (α β : Type) [LE α] [LE β] where
  toFun : α → β
  le_of_le : ∀ a a', a ≤ a' → toFun a ≤ toFun a'

@[ext]
structure OrderPresMonoidHom (M N : Type) [Monoid M] [LE M] [Monoid N] [LE N] extends
MonoidHom₁ M N, OrderPresHom M N

class OrderPresHomClass (F : Type) (α β : outParam Type) [LE α] [LE β]

instance (α β : Type) [LE α] [LE β] : OrderPresHomClass (OrderPresHom α β) α β where

instance (α β : Type) [LE α] [Monoid α] [LE β] [Monoid β] :
    OrderPresHomClass (OrderPresMonoidHom α β) α β where

instance (α β : Type) [LE α] [Monoid α] [LE β] [Monoid β] :
    MonoidHomClass₃ (OrderPresMonoidHom α β) α β
  := sorry

8.3. 子对象

定义了一些代数结构及其态射之后，下一步是考虑继承这些代数结构的集合，例如子群或子环。这与前一个主题有很大重叠。事实上，X 中的集合实现为从 X 到 Prop 的函数，因此子对象就是满足某个谓词的函数。于是可以复用许多导致 DFunLike 类及其后代的思想。我们不会复用 DFunLike 本身，因为这会打破从 Set X 到 X → Prop 的抽象屏障。相反，有一个 SetLike 类。这个类不是把一个到函数类型的嵌入打包起来，而是把一个到 Set 类型的嵌入打包起来，并定义相应的强制转换和 Membership 实例。

@[ext]
structure Submonoid₁ (M : Type) [Monoid M] where
  /-- The carrier of a submonoid. -/
  carrier : Set M
  /-- The product of two elements of a submonoid belongs to the submonoid. -/
  mul_mem {a b} : a ∈ carrier → b ∈ carrier → a * b ∈ carrier
  /-- The unit element belongs to the submonoid. -/
  one_mem : 1 ∈ carrier

/-- Submonoids in `M` can be seen as sets in `M`. -/
instance [Monoid M] : SetLike (Submonoid₁ M) M where
  coe := Submonoid₁.carrier
  coe_injective' _ _ := Submonoid₁.ext

有了上面的 SetLike 实例，我们已经可以自然地陈述子幺半群 N 包含 1，而不必使用 N.carrier。也可以默默地把 N 当作 M 中的集合，并取它在某个映射下的直接像。

example [Monoid M] (N : Submonoid₁ M) : 1 ∈ N := N.one_mem

example [Monoid M] (N : Submonoid₁ M) (α : Type) (f : M → α) := f '' N

我们还有一个使用 Subtype 的到 Type 的强制转换。因此，给定子幺半群 N，可以写参数 (x : N)，它可以被强制转换为属于 N 的 M 的元素。

example [Monoid M] (N : Submonoid₁ M) (x : N) : (x : M) ∈ N := x.property

使用这个到 Type 的强制转换，还可以处理给子幺半群配备幺半群结构的任务。我们将使用上面与 N 相关的类型给出的强制转换，以及断言该强制转换为单射的引理 SetCoe.ext。二者都由 SetLike 实例提供。

instance SubMonoid₁Monoid [Monoid M] (N : Submonoid₁ M) : Monoid N where
  mul := fun x y ↦ ⟨x*y, N.mul_mem x.property y.property⟩
  mul_assoc := fun x y z ↦ SetCoe.ext (mul_assoc (x : M) y z)
  one := ⟨1, N.one_mem⟩
  one_mul := fun x ↦ SetCoe.ext (one_mul (x : M))
  mul_one := fun x ↦ SetCoe.ext (mul_one (x : M))

注意，在上面的实例中，除了使用到 M 的强制转换并调用 property 字段之外，也可以如下使用解构绑定器。

example [Monoid M] (N : Submonoid₁ M) : Monoid N where
  mul := fun ⟨x, hx⟩ ⟨y, hy⟩ ↦ ⟨x*y, N.mul_mem hx hy⟩
  mul_assoc := fun ⟨x, _⟩ ⟨y, _⟩ ⟨z, _⟩ ↦ SetCoe.ext (mul_assoc x y z)
  one := ⟨1, N.one_mem⟩
  one_mul := fun ⟨x, _⟩ ↦ SetCoe.ext (one_mul x)
  mul_one := fun ⟨x, _⟩ ↦ SetCoe.ext (mul_one x)

为了把关于子幺半群的引理应用到子群或子环上，需要一个类，就像态射的情形一样。注意，这个类把 SetLike 实例作为参数，因此不需要载体字段，并且可以在其字段中使用成员关系记号。

class SubmonoidClass₁ (S : Type) (M : Type) [Monoid M] [SetLike S M] : Prop where
  mul_mem : ∀ (s : S) {a b : M}, a ∈ s → b ∈ s → a * b ∈ s
  one_mem : ∀ s : S, 1 ∈ s

instance [Monoid M] : SubmonoidClass₁ (Submonoid₁ M) M where
  mul_mem := Submonoid₁.mul_mem
  one_mem := Submonoid₁.one_mem

作为练习，你应定义一个 Subgroup₁ 结构，为它配备 SetLike 实例和 SubmonoidClass₁ 实例，在与 Subgroup₁ 相关的子类型上放置 Group 实例，并定义一个 SubgroupClass₁ 类。

关于 Mathlib 中给定代数对象的子对象，还有一个非常重要的事实：它们总是形成完备格，而且这个结构经常被使用。例如，你可能想寻找“子幺半群的交仍是子幺半群”的引理。但它不会是一个引理，而会是一个下确界构造。我们来处理两个子幺半群的情形。

instance [Monoid M] : Min (Submonoid₁ M) :=
  ⟨fun S₁ S₂ ↦
    { carrier := S₁ ∩ S₂
      one_mem := ⟨S₁.one_mem, S₂.one_mem⟩
      mul_mem := fun ⟨hx, hx'⟩ ⟨hy, hy'⟩ ↦ ⟨S₁.mul_mem hx hy, S₂.mul_mem hx' hy'⟩ }⟩

这使我们可以把两个子幺半群的交作为子幺半群得到。

example [Monoid M] (N P : Submonoid₁ M) : Submonoid₁ M := N ⊓ P

你可能觉得可惜，上面的例子中必须使用下确界符号 ⊓，而不是交符号 ∩。但请想一想上确界。两个子幺半群的并不是子幺半群。然而，子幺半群仍然形成格（甚至完备格）。实际上，N ⊔ P 是由 N 与 P 的并生成的子幺半群，当然若把它记为 N ∪ P 会非常令人困惑。所以可以看出，使用 N ⊓ P 要一致得多。它在各种代数结构之间也一致得多。一开始看到两个向量子空间 E 和 F 的和记为 E ⊔ F 而不是 E + F，可能有些奇怪。但你会习惯的。很快你会觉得记号 E + F 反而是一种干扰，因为它强调 E ⊔ F 的元素可以写成 E 中一个元素与 F 中一个元素之和这一轶事性事实，而不是强调 E ⊔ F 是同时包含 E 和 F 的最小向量子空间这一根本事实。

本章最后一个主题是商。我们仍想解释 Mathlib 中如何构造方便记号并避免代码重复。这里的主要装置是类 HasQuotient，它允许类似 M ⧸ N 的记号。注意，商符号 ⧸ 是一个特殊 Unicode 字符，不是普通 ASCII 除号。

作为例子，我们将构造交换幺半群对一个子幺半群的商，并把证明留给你。在最后一个例子中，可以使用 Setoid.refl，但它不会自动拾取相关的 Setoid 结构。可以用 @ 语法提供所有参数来修正这个问题，例如 @Setoid.refl M N.Setoid。

def Submonoid.Setoid [CommMonoid M] (N : Submonoid M) : Setoid M  where
  r := fun x y ↦ ∃ w ∈ N, ∃ z ∈ N, x*w = y*z
  iseqv := {
    refl := fun x ↦ ⟨1, N.one_mem, 1, N.one_mem, rfl⟩
    symm := fun ⟨w, hw, z, hz, h⟩ ↦ ⟨z, hz, w, hw, h.symm⟩
    trans := by
      sorry
  }

instance [CommMonoid M] : HasQuotient M (Submonoid M) where
  Quotient := fun N ↦ Quotient N.Setoid

def QuotientMonoid.mk [CommMonoid M] (N : Submonoid M) : M → M ⧸ N := Quotient.mk N.Setoid

instance [CommMonoid M] (N : Submonoid M) : Monoid (M ⧸ N) where
  mul := Quotient.map₂ (· * ·) (by
      sorry
        )
  mul_assoc := by
      sorry
  one := QuotientMonoid.mk N 1
