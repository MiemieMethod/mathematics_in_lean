3. 逻辑

上一章中，我们处理了等式、不等式，以及诸如“\(x\) 整除 \(y\)”这样的基本数学命题。复杂的数学命题由这类简单命题借助逻辑术语构成，例如“且”“或”“非”，以及“若……则……”“每个”和“存在某个”。本章将说明如何处理以这种方式构成的命题。

3.1. 蕴含与全称量词

考虑 #check 之后的命题：

#check ∀ x : ℝ, 0 ≤ x → |x| = x

用文字来说，我们会说：“对每个实数 x，若 0 ≤ x，则 x 的绝对值等于 x。”我们也可以有更复杂的命题，例如：

#check ∀ x y ε : ℝ, 0 < ε → ε ≤ 1 → |x| < ε → |y| < ε → |x * y| < ε

用文字来说，我们会说：“对每个 x、y 和 ε，若 0 < ε ≤ 1，x 的绝对值小于 ε，且 y 的绝对值小于 ε，则 x * y 的绝对值小于 ε。”在 Lean 中，一串蕴含中的隐含括号向右结合。因此，上面的表达式意为：“若 0 < ε，则若 ε ≤ 1，则若 |x| < ε，……”于是，该表达式表示所有假设合在一起推出结论。

你已经看到，尽管这个命题中的全称量词遍历对象，而蕴含箭头引入假设，Lean 对二者的处理方式非常相似。特别地，如果你证明了这种形式的定理，就可以用同样方式把它应用于对象和假设。我们将以下命题作为例子；稍后会帮助你证明它：

theorem my_lemma : ∀ x y ε : ℝ, 0 < ε → ε ≤ 1 → |x| < ε → |y| < ε → |x * y| < ε :=
  sorry

section
variable (a b δ : ℝ)
variable (h₀ : 0 < δ) (h₁ : δ ≤ 1)
variable (ha : |a| < δ) (hb : |b| < δ)

#check my_lemma a b δ
#check my_lemma a b δ h₀ h₁
#check my_lemma a b δ h₀ h₁ ha hb

end

你也已经看到，在 Lean 中，当量化变量可以由后续假设推断出来时，常常使用花括号把它们设为隐式变量。这样做之后，应用引理时只需把它应用于假设，而不必提及那些对象。

theorem my_lemma2 : ∀ {x y ε : ℝ}, 0 < ε → ε ≤ 1 → |x| < ε → |y| < ε → |x * y| < ε :=
  sorry

section
variable (a b δ : ℝ)
variable (h₀ : 0 < δ) (h₁ : δ ≤ 1)
variable (ha : |a| < δ) (hb : |b| < δ)

#check my_lemma2 h₀ h₁ ha hb

end

到此为止，你也知道：若使用 apply 策略把 my_lemma 应用于形如 |a * b| < δ 的目标，就会留下若干新目标，要求你证明每一个假设。

要证明这样的命题，可以使用 intro 策略。请看它在这个例子中的作用：

theorem my_lemma3 :
    ∀ {x y ε : ℝ}, 0 < ε → ε ≤ 1 → |x| < ε → |y| < ε → |x * y| < ε := by
  intro x y ε epos ele1 xlt ylt
  sorry

对于全称量化的变量，我们可以使用任意名称；它们不必叫作 x、y 和 ε。注意，即使这些变量被标记为隐式，我们仍然必须引入它们：把它们设为隐式，意味着在写使用 my_lemma 的表达式时省略它们；但它们仍是我们正在证明的命题的本质组成部分。在 intro 命令之后，目标就变成了如果我们像上一节那样在冒号之前列出所有变量和假设时一开始会看到的目标。稍后我们会看到，为什么有时必须在证明开始之后再引入变量和假设。

为了帮助你证明这个引理，我们先给出开头部分：

theorem my_lemma4 :
    ∀ {x y ε : ℝ}, 0 < ε → ε ≤ 1 → |x| < ε → |y| < ε → |x * y| < ε := by
  intro x y ε epos ele1 xlt ylt
  calc
    |x * y| = |x| * |y| := sorry
    _ ≤ |x| * ε := sorry
    _ < 1 * ε := sorry
    _ = ε := sorry

请使用定理 abs_mul、mul_le_mul、abs_nonneg、mul_lt_mul_of_pos_right 和 one_mul 完成证明。记住，可以用 Ctrl-space 补全（Mac 上为 Cmd-space 补全）寻找这类定理。还要记住，可以使用 .mp 与 .mpr，或 .1 与 .2，从一个当且仅当命题中取出两个方向。

全称量词常常隐藏在定义之中；必要时，Lean 会展开定义以显露它们。例如，让我们定义两个谓词 FnUb f a 和 FnLb f a，其中 f 是从实数到实数的函数，a 是实数。前者表示 a 是 f 的取值的上界，后者表示 a 是 f 的取值的下界。

def FnUb (f : ℝ → ℝ) (a : ℝ) : Prop :=
  ∀ x, f x ≤ a

def FnLb (f : ℝ → ℝ) (a : ℝ) : Prop :=
  ∀ x, a ≤ f x

在下一个例子中，fun x ↦ f x + g x 是把 x 映到 f x + g x 的函数。在类型论中，从表达式 f x + g x 到这个函数的过程称为 lambda 抽象。

example (hfa : FnUb f a) (hgb : FnUb g b) : FnUb (fun x ↦ f x + g x) (a + b) := by
  intro x
  dsimp
  apply add_le_add
  apply hfa
  apply hgb

对目标 FnUb (fun x ↦ f x + g x) (a + b) 应用 intro，会迫使 Lean 展开 FnUb 的定义，并为全称量词引入 x。此时目标变为 (fun (x : ℝ) ↦ f x + g x) x ≤ a + b。但是，把 (fun x ↦ f x + g x) 应用于 x 应得到 f x + g x，而 dsimp 命令执行这一化简。（其中 “d” 表示 “definitional”。）删除该命令后证明仍然成立；Lean 无论如何都必须执行这个约简，才能理解下一个 apply。dsimp 只是让目标更可读，并帮助我们判断下一步该做什么。另一种选择是使用 change 策略，写成 change f x + g x ≤ a + b。这有助于提高证明的可读性，并使你能更好地控制目标如何变形。

证明的其余部分是例行的。最后两个 apply 命令迫使 Lean 展开假设中 FnUb 的定义。请尝试以类似方式完成下面这些证明：

example (hfa : FnLb f a) (hgb : FnLb g b) : FnLb (fun x ↦ f x + g x) (a + b) :=
  sorry

example (nnf : FnLb f 0) (nng : FnLb g 0) : FnLb (fun x ↦ f x * g x) 0 :=
  sorry

example (hfa : FnUb f a) (hgb : FnUb g b) (nng : FnLb g 0) (nna : 0 ≤ a) :
    FnUb (fun x ↦ f x * g x) (a * b) :=
  sorry

尽管我们为从实数到实数的函数定义了 FnUb 和 FnLb，但你应当认识到，这些定义和证明要一般得多。只要函数的陪域带有某种序结构，这些定义就适用于任意两个类型之间的函数。检查定理 add_le_add 的类型会发现，它适用于任何与相关序相容的幺半群结构；这究竟是什么意思目前并不重要，但值得知道自然数、整数、有理数和实数都是其实例。因此，如果我们在这个一般性层次证明定理 fnUb_add，它就可用于所有这些实例。

variable {α : Type*} {R : Type*} [AddCommMonoid R] [PartialOrder R] [IsOrderedCancelAddMonoid R]

#check add_le_add

def FnUb' (f : α → R) (a : R) : Prop :=
  ∀ x, f x ≤ a

theorem fnUb_add {f g : α → R} {a b : R} (hfa : FnUb' f a) (hgb : FnUb' g b) :
    FnUb' (fun x ↦ f x + g x) (a + b) := fun x ↦ add_le_add (hfa x) (hgb x)

你已经在 第 2.2 节 中见过这样的方括号，尽管我们尚未解释它们的含义。为了具体起见，本书多数例子会坚持使用实数；但值得知道的是，Mathlib 包含许多在高度一般性下工作的定义和定理。

作为隐藏全称量词的另一个例子，Mathlib 定义了谓词 Monotone，表示函数关于其参数单调不减：

example (f : ℝ → ℝ) (h : Monotone f) : ∀ {a b}, a ≤ b → f a ≤ f b :=
  @h

性质 Monotone f 按定义恰好就是冒号之后的表达式。我们需要在 h 前写符号 @；否则 Lean 会展开 h 的隐式参数并插入占位符。

证明关于单调性的命题时，通常使用 intro 引入两个变量，例如 a 和 b，以及假设 a ≤ b。要使用单调性假设，可以把它应用于合适的参数和假设，然后把得到的表达式应用于目标。也可以把它应用于目标，让 Lean 通过把剩余假设显示为新的子目标来帮助你反向推理。

example (mf : Monotone f) (mg : Monotone g) : Monotone fun x ↦ f x + g x := by
  intro a b aleb
  apply add_le_add
  apply mf aleb
  apply mg aleb

当证明如此简短时，直接给出证明项往往很方便。为了描述一个临时引入对象 a、b 和假设 aleb 的证明，Lean 使用记号 fun a b aleb ↦ ...。这类似于表达式 fun x ↦ x^2 通过临时命名一个对象 x，然后用它描述一个值来定义函数。因此，前一个证明中的 intro 命令对应下一个证明项中的 lambda 抽象，而 apply 命令则对应构造定理对其参数的应用。

example (mf : Monotone f) (mg : Monotone g) : Monotone fun x ↦ f x + g x :=
  fun _a _b aleb ↦ add_le_add (mf aleb) (mg aleb)

这里有一个有用技巧：如果你开始写证明项 fun a b aleb ↦ _，用下划线表示剩余表达式应出现的位置，Lean 会标出错误，说明它无法猜出该表达式的值。若在 VS Code 中查看 Lean InfoView 窗口，或把鼠标悬停在波浪线错误标记上，Lean 会显示剩余表达式必须解决的目标。

你会注意到，Lean 会对上述证明打印警告，说 a 和 b 未在函数体中使用。这是一种通用机制，例如可用于避免定理中含有不必要的假设。这里它并没有告诉我们什么真正有用的信息。我们可以写 set_option linter.unusedVariables false，将这个警告禁用到当前小节结束；也可以只在该例子中禁用，在例子上方写 set_option linter.unusedVariables false in。

我们也可以把 a 和 b 替换为下划线，告诉 Lean 不想给它们命名。Lean 随后会使用自动生成的不可访问名称，例如 x✝。另一种选择是在名称前加下划线，告诉 Lean 我们并不打算使用它们。

请尝试用策略或证明项证明下面的例子：

example {c : ℝ} (mf : Monotone f) (nnc : 0 ≤ c) : Monotone fun x ↦ c * f x :=
  sorry

example (mf : Monotone f) (mg : Monotone g) : Monotone fun x ↦ f (g x) :=
  sorry

下面还有一些例子。从 \(\Bbb R\) 到 \(\Bbb R\) 的函数 \(f\)，若对每个 \(x\) 都有 \(f(-x) = f(x)\)，则称为偶函数；若对每个 \(x\) 都有 \(f(-x) = -f(x)\)，则称为奇函数。下面的例子形式化定义这两个概念，并证明其中一个事实。你可以完成其余证明。

def FnEven (f : ℝ → ℝ) : Prop :=
  ∀ x, f x = f (-x)

def FnOdd (f : ℝ → ℝ) : Prop :=
  ∀ x, f x = -f (-x)

example (ef : FnEven f) (eg : FnEven g) : FnEven fun x ↦ f x + g x := by
  intro x
  calc
    (fun x ↦ f x + g x) x = f x + g x := rfl
    _ = f (-x) + g (-x) := by rw [ef, eg]


example (of : FnOdd f) (og : FnOdd g) : FnEven fun x ↦ f x * g x := by
  sorry

example (ef : FnEven f) (og : FnOdd g) : FnOdd fun x ↦ f x * g x := by
  sorry

example (ef : FnEven f) (og : FnOdd g) : FnEven fun x ↦ f (g x) := by
  sorry

第一个证明可以用 dsimp 或 change 缩短，以消除 lambda 抽象。但你可以检查，如果不显式消除 lambda 抽象，后续 rw 将无法工作，因为它无法在表达式中找到模式 f x 和 g x。与某些其他策略不同，rw 在句法层面操作；它不会替你展开定义或进行约简（它有一个变体 erw，会在这个方向上稍微多尝试一点，但也不会多很多）。

一旦知道如何识别隐式全称量词，就会发现它们无处不在。

Mathlib 包含一个很好的集合操作库。回忆一下，Lean 并不使用基于集合论的基础，因此这里“集合”一词具有日常数学意义：给定某个类型 α 的数学对象所组成的一个汇集。若 x 的类型是 α，而 s 的类型是 Set α，则 x ∈ s 是一个命题，断言 x 是 s 的元素。若 y 具有某个不同类型 β，则表达式 y ∈ s 没有意义。这里“没有意义”指的是“没有类型，因而 Lean 不接受它作为良构命题”。这与 Zermelo-Fraenkel 集合论形成对比；在后者中，对任意数学对象 a 和 b，a ∈ b 都是良构命题。例如，sin ∈ cos 在 ZF 中是良构命题。集合论基础的这一缺陷，是证明助手不采用它的重要动机之一：证明助手应通过发现无意义表达式来帮助我们。在 Lean 中，sin 的类型是 ℝ → ℝ，cos 的类型也是 ℝ → ℝ，即使展开定义后也不等于 Set (ℝ → ℝ)，所以命题 sin ∈ cos 没有意义。人们也可以用 Lean 研究集合论本身。例如，连续统假设相对于 Zermelo-Fraenkel 公理的独立性已经在 Lean 中形式化。但这种集合论元理论完全超出了本书范围。

若 s 和 t 的类型是 Set α，则子集关系 s ⊆ t 定义为 ∀ {x : α}, x ∈ s → x ∈ t。量词中的变量被标记为隐式变量，因此给定 h : s ⊆ t 和 h' : x ∈ s 时，可以写 h h' 作为 x ∈ t 的证明。下面的例子给出了子集关系自反性的策略证明和证明项，并要求你对传递性做同样的事情。

variable {α : Type*} (r s t : Set α)

example : s ⊆ s := by
  intro x xs
  exact xs

theorem Subset.refl : s ⊆ s := fun _x xs ↦ xs

theorem Subset.trans : r ⊆ s → s ⊆ t → r ⊆ t := by
  sorry

正如我们为函数定义 FnUb 一样，也可以定义 SetUb s a，表示 a 是集合 s 的上界；这里假定 s 是某个带序类型中元素组成的集合。在下一个例子中，我们要求你证明：如果 a 是 s 的上界且 a ≤ b，那么 b 也是 s 的上界。

variable {α : Type*} [PartialOrder α]
variable (s : Set α) (a b : α)

def SetUb (s : Set α) (a : α) :=
  ∀ x, x ∈ s → x ≤ a

example (h : SetUb s a) (h' : a ≤ b) : SetUb s b :=
  sorry

本节最后看一个重要例子。若对每个 \(x_1\) 和 \(x_2\)，由 \(f(x_1) = f(x_2)\) 可推出 \(x_1 = x_2\)，则称函数 \(f\) 是单射。Mathlib 定义了 Function.Injective f，其中 x₁ 和 x₂ 为隐式参数。下一个例子说明，在实数上，任何加上常数的函数都是单射。随后我们要求你证明，乘以非零常数的函数也是单射；可以从例子中的引理名获得启发。记住，在猜出引理名开头之后，应使用 Ctrl-space 补全。

open Function

example (c : ℝ) : Injective fun x ↦ x + c := by
  intro x₁ x₂ h'
  exact (add_left_inj c).mp h'

example {c : ℝ} (h : c ≠ 0) : Injective fun x ↦ c * x := by
  sorry

最后，请证明两个单射函数的复合仍为单射：

variable {α : Type*} {β : Type*} {γ : Type*}
variable {g : β → γ} {f : α → β}

example (injg : Injective g) (injf : Injective f) : Injective fun x ↦ g (f x) := by
  sorry

3.2. 存在量词

存在量词可以在 VS Code 中输入为 \ex，用于表示“存在”。Lean 中的形式表达式 ∃ x : ℝ, 2 < x ∧ x < 3 表示在 2 与 3 之间存在一个实数。（我们将在 第 3.4 节 中讨论合取符号 ∧。）证明这类命题的标准方式，是给出一个实数，并证明它具有所陈述的性质。数 2.5 具有所需性质；它可以输入为 5 / 2，或在 Lean 无法从语境推断我们指的是实数时输入为 (5 : ℝ) / 2；norm_num 策略可以证明它满足该描述。

有几种方式可以把这些信息组合起来。若目标以存在量词开头，use 策略用于给出对象，并留下证明该对象满足相应性质的目标。

example : ∃ x : ℝ, 2 < x ∧ x < 3 := by
  use 5 / 2
  norm_num

除了数据之外，也可以向 use 策略给出证明：

example : ∃ x : ℝ, 2 < x ∧ x < 3 := by
  have h1 : 2 < (5 : ℝ) / 2 := by norm_num
  have h2 : (5 : ℝ) / 2 < 3 := by norm_num
  use 5 / 2, h1, h2

事实上，use 策略还会自动尝试使用可用假设。

example : ∃ x : ℝ, 2 < x ∧ x < 3 := by
  have h : 2 < (5 : ℝ) / 2 ∧ (5 : ℝ) / 2 < 3 := by norm_num
  use 5 / 2

另一种方式是使用 Lean 的匿名构造子记号来构造存在量词的证明。

example : ∃ x : ℝ, 2 < x ∧ x < 3 :=
  have h : 2 < (5 : ℝ) / 2 ∧ (5 : ℝ) / 2 < 3 := by norm_num
  ⟨5 / 2, h⟩

注意，这里没有 by；我们给出的是显式证明项。左右尖括号可分别输入为 \< 和 \>，它们告诉 Lean 使用适合当前目标的构造方式，把给定数据组合起来。我们也可以不先进入策略模式而直接使用这个记号：

example : ∃ x : ℝ, 2 < x ∧ x < 3 :=
  ⟨5 / 2, by norm_num⟩

现在我们知道如何证明存在命题。那么如何使用存在命题呢？如果已知存在一个具有某性质的对象，就应当能够给其中任意一个命名并对它推理。例如，回忆上一节中的谓词 FnUb f a 和 FnLb f a，它们分别表示 a 是 f 的上界或下界。我们可以用存在量词说“f 有界”，而不指明具体界：

def FnUb (f : ℝ → ℝ) (a : ℝ) : Prop :=
  ∀ x, f x ≤ a

def FnLb (f : ℝ → ℝ) (a : ℝ) : Prop :=
  ∀ x, a ≤ f x

def FnHasUb (f : ℝ → ℝ) :=
  ∃ a, FnUb f a

def FnHasLb (f : ℝ → ℝ) :=
  ∃ a, FnLb f a

我们可以使用上一节中的定理 FnUb_add 证明：若 f 和 g 都有上界，则 fun x ↦ f x + g x 也有上界。

variable {f g : ℝ → ℝ}

example (ubf : FnHasUb f) (ubg : FnHasUb g) : FnHasUb fun x ↦ f x + g x := by
  rcases ubf with ⟨a, ubfa⟩
  rcases ubg with ⟨b, ubgb⟩
  use a + b
  apply fnUb_add ubfa ubgb

rcases 策略会拆开存在量词中的信息。像 ⟨a, ubfa⟩ 这样的标注使用与匿名构造子相同的尖括号，称为模式；它描述了我们在拆开主参数时预期得到的信息。给定假设 ubf，表示 f 有上界，命令 rcases ubf with ⟨a, ubfa⟩ 会向语境加入一个新变量 a，表示某个上界，同时加入假设 ubfa，表示它具有给定性质。目标保持不变；改变的是我们现在可以使用这个新对象和新假设来证明目标。这是数学推理中的常见方法：我们拆开由某个假设断言或蕴含其存在性的对象，然后用它建立其他事物的存在性。

请尝试使用这种方法建立下面的命题。你可能会发现，把上一节中的某些例子像 fn_ub_add 那样改成命名定理会很有用；也可以直接把参数插入证明中。

example (lbf : FnHasLb f) (lbg : FnHasLb g) : FnHasLb fun x ↦ f x + g x := by
  sorry

example {c : ℝ} (ubf : FnHasUb f) (h : c ≥ 0) : FnHasUb fun x ↦ c * f x := by
  sorry

rcases 中的 “r” 表示 “recursive”，因为它允许我们使用任意复杂的模式来拆开嵌套数据。rintro 策略是 intro 与 rcases 的组合：

example : FnHasUb f → FnHasUb g → FnHasUb fun x ↦ f x + g x := by
  rintro ⟨a, ubfa⟩ ⟨b, ubgb⟩
  exact ⟨a + b, fnUb_add ubfa ubgb⟩

事实上，Lean 也支持在表达式和证明项中使用带模式匹配的 fun：

example : FnHasUb f → FnHasUb g → FnHasUb fun x ↦ f x + g x :=
  fun ⟨a, ubfa⟩ ⟨b, ubgb⟩ ↦ ⟨a + b, fnUb_add ubfa ubgb⟩

拆开假设中的信息是如此重要，以至于 Lean 和 Mathlib 提供了许多方法。例如，obtain 策略提供了富有提示性的语法：

example (ubf : FnHasUb f) (ubg : FnHasUb g) : FnHasUb fun x ↦ f x + g x := by
  obtain ⟨a, ubfa⟩ := ubf
  obtain ⟨b, ubgb⟩ := ubg
  exact ⟨a + b, fnUb_add ubfa ubgb⟩

可以把第一条 obtain 指令理解为：把 ubf 的“内容”与给定模式匹配，并把各组成部分赋给命名变量。我们说 rcases 和 obtain 会析构它们的参数。

Lean 还支持类似其他函数式编程语言中的语法：

example (ubf : FnHasUb f) (ubg : FnHasUb g) : FnHasUb fun x ↦ f x + g x := by
  cases ubf
  case intro a ubfa =>
    cases ubg
    case intro b ubgb =>
      exact ⟨a + b, fnUb_add ubfa ubgb⟩

example (ubf : FnHasUb f) (ubg : FnHasUb g) : FnHasUb fun x ↦ f x + g x := by
  cases ubf
  next a ubfa =>
    cases ubg
    next b ubgb =>
      exact ⟨a + b, fnUb_add ubfa ubgb⟩

example (ubf : FnHasUb f) (ubg : FnHasUb g) : FnHasUb fun x ↦ f x + g x := by
  match ubf, ubg with
    | ⟨a, ubfa⟩, ⟨b, ubgb⟩ =>
      exact ⟨a + b, fnUb_add ubfa ubgb⟩

example (ubf : FnHasUb f) (ubg : FnHasUb g) : FnHasUb fun x ↦ f x + g x :=
  match ubf, ubg with
    | ⟨a, ubfa⟩, ⟨b, ubgb⟩ =>
      ⟨a + b, fnUb_add ubfa ubgb⟩

在第一个例子中，如果把光标放在 cases ubf 之后，会看到该策略产生一个目标，Lean 给它标记为 intro。（这个具体名称来自用于构造存在命题证明的公理化原语的内部名称。）随后 case 策略为各组成部分命名。第二个例子类似，只是使用 next 而非 case，从而可以避免提及 intro。最后两个例子中的 match 一词强调了这里所做的正是计算机科学家所谓的“模式匹配”。注意，第三个证明以 by 开始；在它之后，策略版本的 match 要求箭头右侧是一个策略证明。最后一个例子是证明项：完全没有策略出现。

在本书余下部分，我们会坚持使用 rcases、rintro 和 obtain，作为使用存在量词的首选方式。不过，了解替代语法并无坏处，尤其是在你可能与计算机科学家共处时。

为了说明 rcases 的一种用法，我们证明一个古老的数学常识：若两个整数 x 和 y 都可以写成两个平方数之和，则它们的乘积 x * y 也可以写成两个平方数之和。事实上，这个命题对任意交换环都成立，而不仅限于整数。在下一个例子中，rcases 一次拆开两个存在量词。然后我们把把 x * y 表示为平方和所需的神奇取值作为列表提供给 use 语句，并用 ring 验证它们有效。

variable {α : Type*} [CommRing α]

def SumOfSquares (x : α) :=
  ∃ a b, x = a ^ 2 + b ^ 2

theorem sumOfSquares_mul {x y : α} (sosx : SumOfSquares x) (sosy : SumOfSquares y) :
    SumOfSquares (x * y) := by
  rcases sosx with ⟨a, b, xeq⟩
  rcases sosy with ⟨c, d, yeq⟩
  rw [xeq, yeq]
  use a * c - b * d, a * d + b * c
  ring

这个证明没有提供太多洞见，但可以用以下方式来说明它的动机。高斯整数是形如 \(a + bi\) 的数，其中 \(a\) 和 \(b\) 是整数，且 \(i = \sqrt{-1}\)。按定义，高斯整数 \(a + bi\) 的范数是 \(a^2 + b^2\)。因此，高斯整数的范数是平方和，且任意平方和都可以这样表示。上述定理反映了这样一个事实：高斯整数乘积的范数等于范数的乘积；如果 \(x\) 是 \(a + bi\) 的范数，而 \(y\) 是 \(c + di\) 的范数，则 \(xy\) 是 \((a + bi)(c + di)\) 的范数。这个晦涩的证明说明，最容易形式化的证明并不总是最清楚的证明。在 第 7.3 节 中，我们会提供定义高斯整数的手段，并用它们给出一个替代证明。

拆开存在量词内部的等式，再用它重写目标中的表达式，这种模式经常出现；频繁到 rcases 策略为它提供了一个缩写：若使用关键字 rfl 代替新的标识符，rcases 会自动执行重写（这个技巧不适用于模式匹配 lambda）。

theorem sumOfSquares_mul' {x y : α} (sosx : SumOfSquares x) (sosy : SumOfSquares y) :
    SumOfSquares (x * y) := by
  rcases sosx with ⟨a, b, rfl⟩
  rcases sosy with ⟨c, d, rfl⟩
  use a * c - b * d, a * d + b * c
  ring

与全称量词一样，只要知道如何识别，存在量词也会隐藏在许多地方。例如，整除关系隐含地就是一个“存在”命题。

example (divab : a ∣ b) (divbc : b ∣ c) : a ∣ c := by
  rcases divab with ⟨d, beq⟩
  rcases divbc with ⟨e, ceq⟩
  rw [ceq, beq]
  use d * e; ring

同样，这为配合 rfl 使用 rcases 提供了一个很好的场景。请在上面的证明中试一试；效果相当顺手！

3.3. 否定

符号 ¬ 用来表示否定，因此 ¬ x < y 表示 x 不小于 y，¬ x = y（或等价地，x ≠ y）表示 x 不等于 y，而 ¬ ∃ z, x < z ∧ z < y 表示不存在严格位于 x 与 y 之间的 z。在 Lean 中，记号 ¬ A 是 A → False 的缩写，可以理解为 A 蕴含矛盾。实际而言，这意味着你已经知道一些处理否定的方法：可以通过引入假设 h : A 并证明 False 来证明 ¬ A；若有 h : ¬ A 和 h' : A，则把 h 应用于 h' 就得到 False。

为说明这一点，考虑严格序的非自反性原理 lt_irrefl，它说对每个 a 都有 ¬ a < a。不对称性原理 lt_asymm 说 a < b → ¬ b < a。我们来证明 lt_asymm 可由 lt_irrefl 推出。

example (h : a < b) : ¬b < a := by
  intro h'
  have : a < a := lt_trans h h'
  apply lt_irrefl a this

这个例子引入了几个新技巧。首先，当你使用 have 而不给出标签时，Lean 会使用名称 this，从而提供一个方便的方式回指它。由于证明很短，我们给出显式证明项。但这个证明中真正应当关注的是 intro 策略的结果：它留下一个 False 目标；以及我们最终通过把 lt_irrefl 应用于 a < a 的证明来证明 False。

下面是另一个例子，它使用上一节定义的谓词 FnHasUb，该谓词表示函数有上界。

example (h : ∀ a, ∃ x, f x > a) : ¬FnHasUb f := by
  intro fnub
  rcases fnub with ⟨a, fnuba⟩
  rcases h a with ⟨x, hx⟩
  have : f x ≤ a := fnuba x
  linarith

请记住，当目标可由语境中的线性等式和不等式推出时，使用 linarith 往往很方便。

  sorry

回忆谓词 Monotone f，它表示 f 单调不减。请使用刚刚列出的某些定理证明如下命题：

我们可以证明，如果把上一段代码中的第一个例子里的 < 替换为 ≤，则该命题无法证明。注意，可以通过给出反例来证明全称量化命题的否定。请完成证明。

example : ¬∀ {f : ℝ → ℝ}, Monotone f → ∀ {a b}, f a ≤ f b → a ≤ b := by
  intro h
  let f := fun x : ℝ ↦ (0 : ℝ)
  have monof : Monotone f := by sorry
  have h' : f 1 ≤ f 0 := le_refl _
  sorry

这个例子引入了 let 策略，它向语境添加一个局部定义。如果把光标放在 let 命令之后，在目标窗口中会看到定义 f : ℝ → ℝ := fun x ↦ 0 已被加入语境。必要时，Lean 会展开 f 的定义。特别地，当我们用 le_refl 证明 f 1 ≤ f 0 时，Lean 会把 f 1 和 f 0 约简为 0。

请使用 le_of_not_gt 证明下面的命题：

example (x : ℝ) (h : ∀ ε > 0, x < ε) : x ≤ 0 := by
  sorry

在我们刚完成的许多证明中，隐含着这样一个事实：若 P 是任意性质，则说没有任何对象具有性质 P，等价于说所有对象都不具有性质 P；而说并非所有对象都具有性质 P，等价于说存在某个对象不具有性质 P。换言之，以下四个蕴含都是有效的（但其中一个不能用目前已经解释的方法证明）：

variable {α : Type*} (P : α → Prop) (Q : Prop)

example (h : ¬∃ x, P x) : ∀ x, ¬P x := by
  sorry

example (h : ∀ x, ¬P x) : ¬∃ x, P x := by
  sorry

example (h : ¬∀ x, P x) : ∃ x, ¬P x := by
  sorry

example (h : ∃ x, ¬P x) : ¬∀ x, P x := by
  sorry

第一、第二和第四个蕴含可以直接用你已经见过的方法证明。我们鼓励你尝试。第三个更困难，因为它要从某个对象不存在会导致矛盾这一事实推出该对象存在。这是经典数学推理的一个实例。我们可以如下用反证法证明第三个蕴含。

example (h : ¬∀ x, P x) : ∃ x, ¬P x := by
  by_contra h'
  apply h
  intro x
  show P x
  by_contra h''
  exact h' ⟨x, h''⟩

请确保理解其工作方式。by_contra 策略允许我们通过假设 ¬ Q 并导出矛盾来证明目标 Q。事实上，它等价于使用等价命题 not_not : ¬ ¬ Q ↔ Q。请确认你可以用 by_contra 证明这个等价的正向方向，而反向方向则由否定的一般规则推出。

example (h : ¬¬Q) : Q := by
  sorry

example (h : Q) : ¬¬Q := by
  sorry

请使用反证法建立下面的命题，它是上面已证明某个蕴含的逆命题。（提示：先使用 intro。）

example (h : ¬FnHasUb f) : ∀ a, ∃ x, f x > a := by
  sorry

处理前面带有否定的复合命题常常很繁琐；数学中常见的做法，是把这类命题替换为等价形式，使否定向内推进。为此，Mathlib 提供了 push_neg 策略，它会以这种方式重述目标（这包括把 ¬ ¬A 化简为 A）。命令 push_neg at h 会重述假设 h。

example (h : ¬∀ a, ∃ x, f x > a) : FnHasUb f := by
  push_neg at h
  exact h

example (h : ¬FnHasUb f) : ∀ a, ∃ x, f x > a := by
  dsimp only [FnHasUb, FnUb] at h
  push_neg at h
  exact h

在第二个例子中，我们用 dsimp 展开 FnHasUb 和 FnUb 的定义。（展开 FnUb 时需要用 dsimp 而不是 rw，因为它出现在量词作用域内。）你可以验证，在上面含有 ¬∃ x, P x 和 ¬∀ x, P x 的例子中，push_neg 策略会得到预期结果。即使还不知道如何使用合取符号，你也应该能够用 push_neg 证明下面的命题：

example (h : ¬Monotone f) : ∃ x y, x ≤ y ∧ f y < f x := by
  sorry

Mathlib 还有一个策略 contrapose，它把目标 A → B 转化为 ¬B → ¬A。类似地，若要从假设 h : A 证明 B，则 contrapose h 会留下一个目标：从假设 ¬B 证明 ¬A。使用 contrapose! 而不是 contrapose，还会同时对目标和相关假设应用 push_neg。

example (h : ¬FnHasUb f) : ∀ a, ∃ x, f x > a := by
  contrapose! h
  exact h

example (x : ℝ) (h : ∀ ε > 0, x ≤ ε) : x ≤ 0 := by
  contrapose! h
  use x / 2
  constructor <;> linarith

我们还没有解释 constructor 命令，也没有解释其后的分号用法；这些将在下一节说明。

本节最后介绍 ex falso 原理，即从矛盾可以推出任何命题。在 Lean 中，这由 False.elim 表示，它对任意命题 P 建立 False → P。这个原理看起来也许奇怪，但经常出现。我们常常通过分情况来证明定理，有时可以证明其中一种情形是矛盾的。在这种情况下，需要断言该矛盾建立了当前目标，以便继续处理下一个情形。（我们将在 第 3.5 节 中看到分情况推理的实例。）

一旦已经得到矛盾，Lean 提供了若干关闭目标的方法。

example (h : 0 < 0) : a > 37 := by
  exfalso
  apply lt_irrefl 0 h

example (h : 0 < 0) : a > 37 :=
  absurd h (lt_irrefl 0)

example (h : 0 < 0) : a > 37 := by
  have h' : ¬0 < 0 := lt_irrefl 0
  contradiction

exfalso 策略把当前目标替换为证明 False 的目标。给定 h : P 和 h' : ¬ P，项 absurd h h' 可以建立任意命题。最后，contradiction 策略会尝试通过在假设中寻找矛盾来关闭目标，例如寻找形如 h : P 与 h' : ¬ P 的一对假设。当然，在这个例子中，linarith 也能工作。

3.4. 合取与当且仅当

你已经看到，合取符号 ∧ 用于表示“且”。constructor 策略允许通过先证明 A 再证明 B，来证明形如 A ∧ B 的命题。

example {x y : ℝ} (h₀ : x ≤ y) (h₁ : ¬y ≤ x) : x ≤ y ∧ x ≠ y := by
  constructor
  · assumption
  intro h
  apply h₁
  rw [h]

在这个例子中，assumption 策略告诉 Lean 寻找一个可以解决目标的假设。注意最后的 rw 通过应用 ≤ 的自反性完成目标。下面是使用匿名构造子尖括号完成前面例子的其他方式。第一个是前一个证明的简洁证明项版本，并在关键字 by 处进入策略模式。

example {x y : ℝ} (h₀ : x ≤ y) (h₁ : ¬y ≤ x) : x ≤ y ∧ x ≠ y :=
  ⟨h₀, fun h ↦ h₁ (by rw [h])⟩

example {x y : ℝ} (h₀ : x ≤ y) (h₁ : ¬y ≤ x) : x ≤ y ∧ x ≠ y :=
  have h : x ≠ y := by
    contrapose! h₁
    rw [h₁]
  ⟨h₀, h⟩

使用合取而不是证明合取时，需要拆开其两个部分的证明。可以使用 rcases 策略，也可以使用 rintro 或模式匹配 fun；其方式都类似于它们在存在量词中的用法。

example {x y : ℝ} (h : x ≤ y ∧ x ≠ y) : ¬y ≤ x := by
  rcases h with ⟨h₀, h₁⟩
  contrapose! h₁
  exact le_antisymm h₀ h₁

example {x y : ℝ} : x ≤ y ∧ x ≠ y → ¬y ≤ x := by
  rintro ⟨h₀, h₁⟩ h'
  exact h₁ (le_antisymm h₀ h')

example {x y : ℝ} : x ≤ y ∧ x ≠ y → ¬y ≤ x :=
  fun ⟨h₀, h₁⟩ h' ↦ h₁ (le_antisymm h₀ h')

与 obtain 策略类似，也存在一种带模式匹配的 have：

example {x y : ℝ} (h : x ≤ y ∧ x ≠ y) : ¬y ≤ x := by
  have ⟨h₀, h₁⟩ := h
  contrapose! h₁
  exact le_antisymm h₀ h₁

与 rcases 不同，这里的 have 策略会把 h 保留在语境中。尽管我们不会使用它们，还是再次给出计算机科学家常用的模式匹配语法：

example {x y : ℝ} (h : x ≤ y ∧ x ≠ y) : ¬y ≤ x := by
  cases h
  case intro h₀ h₁ =>
    contrapose! h₁
    exact le_antisymm h₀ h₁

example {x y : ℝ} (h : x ≤ y ∧ x ≠ y) : ¬y ≤ x := by
  cases h
  next h₀ h₁ =>
    contrapose! h₁
    exact le_antisymm h₀ h₁

example {x y : ℝ} (h : x ≤ y ∧ x ≠ y) : ¬y ≤ x := by
  match h with
    | ⟨h₀, h₁⟩ =>
        contrapose! h₁
        exact le_antisymm h₀ h₁

与存在量词不同，对于假设 h : A ∧ B，也可以写 h.left 和 h.right 来取出两个组成部分的证明；等价地，也可以写 h.1 和 h.2。

example {x y : ℝ} (h : x ≤ y ∧ x ≠ y) : ¬y ≤ x := by
  intro h'
  apply h.right
  exact le_antisymm h.left h'

example {x y : ℝ} (h : x ≤ y ∧ x ≠ y) : ¬y ≤ x :=
  fun h' ↦ h.right (le_antisymm h.left h')

3.5. 析取

证明析取 A ∨ B 的标准方式，是证明 A 或证明 B。left 策略选择 A，而 right 策略选择 B。

variable {x y : ℝ}

example (h : y > x ^ 2) : y > 0 ∨ y < -1 := by
  left
  linarith [pow_two_nonneg x]

example (h : -y > x ^ 2 + 1) : y > 0 ∨ y < -1 := by
  right
  linarith [pow_two_nonneg x]

我们不能用匿名构造子来构造“或”的证明，因为 Lean 必须猜测我们要证明哪一个析取支。写证明项时，可以改用 Or.inl 和 Or.inr 来显式作出选择。这里 inl 是 “introduction left” 的缩写，inr 是 “introduction right” 的缩写。

example (h : y > 0) : y > 0 ∨ y < -1 :=
  Or.inl h

example (h : y < -1) : y > 0 ∨ y < -1 :=
  Or.inr h

通过证明某一边来证明析取，可能看起来有些奇怪。实践中，哪种情形成立通常取决于假设和数据中隐含或显式的情形区分。rcases 策略允许我们使用形如 A ∨ B 的假设。不同于在合取或存在量词中使用 rcases，这里 rcases 会产生两个目标。二者有相同结论，但第一种情形中假定 A 为真，第二种情形中假定 B 为真。换言之，正如其名所示，rcases 策略执行分情况证明。照常，我们可以告诉 Lean 使用什么名称来命名这些假设。在下一个例子中，我们告诉 Lean 在每个分支都使用名称 h。

example : x < |y| → x < y ∨ x < -y := by
  rcases le_or_gt 0 y with h | h
  · rw [abs_of_nonneg h]
    intro h; left; exact h
  · rw [abs_of_neg h]
    intro h; right; exact h

注意，在合取情形中模式从 ⟨h₀, h₁⟩ 变成析取情形中的 h₀ | h₁。可以把第一个模式理解为匹配同时包含 h₀ 和 h₁ 的数据；而第二个带竖线的模式匹配包含 h₀ 或 h₁ 的数据。在这个例子中，由于两个目标彼此分离，我们选择在每种情形中使用同一名称 h。

绝对值函数的定义方式使我们能够立即证明：x ≥ 0 蕴含 |x| = x（这是定理 abs_of_nonneg），而 x < 0 蕴含 |x| = -x（这是 abs_of_neg）。表达式 le_or_gt 0 x 建立 0 ≤ x ∨ x < 0，使我们可以按这两种情形拆分。

Lean 也支持用于析取的计算机科学式模式匹配语法。现在 cases 策略更有吸引力，因为它允许我们命名每个 case，并在更接近使用位置的地方命名引入的假设。

example : x < |y| → x < y ∨ x < -y := by
  cases le_or_gt 0 y
  case inl h =>
    rw [abs_of_nonneg h]
    intro h; left; exact h
  case inr h =>
    rw [abs_of_neg h]
    intro h; right; exact h

名称 inl 和 inr 分别是 “intro left” 和 “intro right” 的缩写。使用 case 的优点是可以按任意顺序证明各情形；Lean 会使用标签找到相关目标。若不关心这一点，可以使用 next、match，甚至使用模式匹配 have。

example : x < |y| → x < y ∨ x < -y := by
  cases le_or_gt 0 y
  next h =>
    rw [abs_of_nonneg h]
    intro h; left; exact h
  next h =>
    rw [abs_of_neg h]
    intro h; right; exact h

example : x < |y| → x < y ∨ x < -y := by
  match le_or_gt 0 y with
    | Or.inl h =>
      rw [abs_of_nonneg h]
      intro h; left; exact h
    | Or.inr h =>
      rw [abs_of_neg h]
      intro h; right; exact h

在 match 的情形中，需要使用证明析取的标准方式的全名 Or.inl 和 Or.inr。本书通常会使用 rcases 来对析取进行分情况讨论。

3.6. 序列与收敛

现在我们掌握了足够多的技巧，可以做一些真正的数学。在 Lean 中，可以把实数序列 \(s_0, s_1, s_2, \ldots\) 表示为函数 s : ℕ → ℝ。若对每个 \(\varepsilon > 0\)，都存在某个位置，从那里往后序列始终位于 \(a\) 的 \(\varepsilon\) 邻域内，也就是说，存在自然数 \(N\)，使得对每个 \(n \ge N\)，都有 \(|s_n - a| < \varepsilon\)，则称该序列收敛到数 \(a\)。在 Lean 中，可以把它写成如下形式：

def ConvergesTo (s : ℕ → ℝ) (a : ℝ) :=
  ∀ ε > 0, ∃ N, ∀ n ≥ N, |s n - a| < ε

记号 ∀ ε > 0, ... 是 ∀ ε, ε > 0 → ... 的便利缩写；类似地，∀ n ≥ N, ... 是 ∀ n, n ≥ N → ... 的缩写。还要记住，ε > 0 又定义为 0 < ε，而 n ≥ N 定义为 N ≤ n。

本节将建立一些关于收敛的性质。但首先，我们讨论三个处理等式时有用的策略。第一个是 ext 策略，它提供了一种证明两个函数相等的方法。设 \(f(x) = x + 1\) 和 \(g(x) = 1 + x\) 是从实数到实数的函数。那么当然有 \(f = g\)，因为它们在每个 \(x\) 处返回相同的值。ext 策略使我们能够通过证明两个函数在所有参数值处取值相同，来证明函数之间的等式。

example : (fun x y : ℝ ↦ (x + y) ^ 2) = fun x y : ℝ ↦ x ^ 2 + 2 * x * y + y ^ 2 := by
  ext
  ring

稍后我们会看到，ext 实际上更加一般，并且可以指定出现的变量名。例如，你可以尝试把上面证明中的 ext 替换为 ext u v。第二个策略 congr 允许我们通过调和两个表达式中不同的部分来证明等式：

example (a b : ℝ) : |a| = |a - b + b| := by
  congr
  ring

这里，congr 策略剥去两边的 abs，留下要证明的目标 a = a - b + b。

最后，convert 策略用于在某个定理的结论与当前目标不完全匹配时，把该定理应用于目标。例如，假设我们想从 1 < a 证明 a < a * a。库中的一个定理 mul_lt_mul_iff_left₀ 可让我们证明 1 * a < a * a。一种做法是反向工作，把目标重写为这种形式。相反，convert 策略允许我们按原样应用该定理，并留下证明若干等式的任务，这些等式正是使目标匹配所需的。

example {a : ℝ} (h : 1 < a) : a < a * a := by
  convert (mul_lt_mul_iff_left₀ _).2 h
  · rw [one_mul]
  exact lt_trans zero_lt_one h

这个例子展示了另一个有用技巧：当我们应用一个带下划线的表达式而 Lean 无法自动填入该下划线时，它会把它简单地留下来，作为另一个目标交给我们。

下面证明任意常值序列 \(a, a, a, \ldots\) 都收敛。

theorem convergesTo_const (a : ℝ) : ConvergesTo (fun _x : ℕ ↦ a) a := by
  intro ε εpos
  use 0
  intro n nge
  rw [sub_self, abs_zero]
  apply εpos

Lean 有一个策略 simp，它常常能省去手工执行类似 rw [sub_self, abs_zero] 步骤的麻烦。我们很快会进一步介绍它。

作为一个更有趣的定理，我们来证明：若 s 收敛到 a，且 t 收敛到 b，则 fun n ↦ s n + t n 收敛到 a + b。在开始写形式化证明之前，心中有一个清楚的纸笔证明会很有帮助。给定大于 0 的 ε，思路是使用假设得到一个 Ns，使得从该点以后，s 与 a 的距离小于 ε / 2；再得到一个 Nt，使得从该点以后，t 与 b 的距离小于 ε / 2。于是，只要 n 大于等于 Ns 和 Nt 的最大值，序列 fun n ↦ s n + t n 就应当与 a + b 的距离小于 ε。下面的例子开始实现这个策略。看看你能否把它完成。

theorem convergesTo_add {s t : ℕ → ℝ} {a b : ℝ}
      (cs : ConvergesTo s a) (ct : ConvergesTo t b) :
    ConvergesTo (fun n ↦ s n + t n) (a + b) := by
  intro ε εpos
  dsimp -- this line is not needed but cleans up the goal a bit.
  have ε2pos : 0 < ε / 2 := by linarith
  rcases cs (ε / 2) ε2pos with ⟨Ns, hs⟩
  rcases ct (ε / 2) ε2pos with ⟨Nt, ht⟩
  use max Ns Nt
  sorry

作为提示，可以使用 le_of_max_le_left 和 le_of_max_le_right，而 norm_num 可以证明 ε / 2 + ε / 2 = ε。此外，使用 congr 策略证明 |s n + t n - (a + b)| 等于 |(s n - a) + (t n - b)| 也很有帮助，因为这样就可以使用三角不等式。注意，我们把所有变量 s、t、a 和 b 都标记为隐式变量，因为它们可以由假设推断出来。

把同一个定理中的加法替换为乘法后，证明会更棘手。我们将先证明一些辅助命题。请看看能否也完成下一个证明，它表明若 s 收敛到 a，则 fun n ↦ c * s n 收敛到 c * a。按 c 是否等于零分情况讨论会很有帮助。我们已经处理了零的情形，并留下了带有额外假设 c 非零时的结果让你证明。

    rw [h]

  rcases ct _ pos₀ with ⟨N₁, h₁⟩
  sorry

如果你已经做到这里，恭喜！我们距离目标定理已经很近了。下面的证明将完成它。

theorem convergesTo_mul {s t : ℕ → ℝ} {a b : ℝ}
      (cs : ConvergesTo s a) (ct : ConvergesTo t b) :
    ConvergesTo (fun n ↦ s n * t n) (a * b) := by
  have h₁ : ConvergesTo (fun n ↦ s n * (t n + -b)) 0 := by
    apply aux cs
    convert convergesTo_add ct (convergesTo_const (-b))
    ring
  have := convergesTo_add h₁ (convergesTo_mul_const b cs)
  convert convergesTo_add h₁ (convergesTo_mul_const b cs) using 1
  · ext; ring
  ring

作为另一个有挑战性的练习，请尝试补全下面的证明草稿，证明极限唯一。（如果你很有信心，可以删除证明草稿并尝试从零开始证明。）

theorem convergesTo_unique {s : ℕ → ℝ} {a b : ℝ}
      (sa : ConvergesTo s a) (sb : ConvergesTo s b) :
    a = b := by
  by_contra abne
  have : |a - b| > 0 := by sorry
  let ε := |a - b| / 2
  have εpos : ε > 0 := by
    change |a - b| / 2 > 0
    linarith
  rcases sa ε εpos with ⟨Na, hNa⟩
  rcases sb ε εpos with ⟨Nb, hNb⟩
  let N := max Na Nb
  have absa : |s N - a| < ε := by sorry
  have absb : |s N - b| < ε := by sorry
  have : |a - b| < |a - b| := by sorry
  exact lt_irrefl _ this

本节最后指出，我们的证明可以推广。例如，我们对自然数所使用的唯一性质，是其结构带有偏序以及 min 和 max。可以检查，如果把各处的 ℕ 替换为任意线性序 α，一切仍然成立：

variable {α : Type*} [LinearOrder α]

def ConvergesTo' (s : α → ℝ) (a : ℝ) :=
  ∀ ε > 0, ∃ N, ∀ n ≥ N, |s n - a| < ε
